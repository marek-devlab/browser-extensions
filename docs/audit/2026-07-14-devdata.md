## (a) Вердикт

**Почти готово, но один архитектурный провал сводит на нет главную обещанную гарантию.** База выстроена честно и на удивление аккуратно: **сети нет вообще** (0 `fetch`/`XHR`/`WebSocket`, 0 `console.*`, 0 `innerHTML`/`eval`), тяжёлое и враждебное реально живёт в Worker'е с настоящим `terminate()` по таймауту (парс 30 с, конверт 30 с, схема 5 с), JSON Schema валидируется `@cfworker` (без `new Function`, под MV3 CSP), внешние `$ref` отклоняются явно, ReDoS-`pattern` заперт в терминабельном потоке. Секреты: **HS256-секрет и приватный/публичный ключ живут только в React-state таба JWT и не пишутся ни в один storage-item** — это проверено по всем путям, включая обработчики ошибок (сообщения об ошибках не эхоят секрет/токен). Разрешения минимальны и все используются (нет Purple Potassium); `scripting` и `<all_urls>` — optional и запрашиваются честно; permission-факт читается из браузера, а не из флага. Single purpose соблюдён: один инструмент с табами, никаких вторых точек входа.

Блокирует одно (🔴 B1): **автодетект формата гоняет `JSON.parse` по всему документу на главном потоке**, до создания воркер-джоба и до появления спиннера/«Отмены». Это ровно тот фриз, ради предотвращения которого построена вся Worker-архитектура, и на дефолтном пути (`defaultFormat: 'auto'`); у верхней границы (`HARD_MAX` 200 МБ) выброшенный впустую граф объектов способен уронить вкладку по памяти — ту самую, которую воркер должен был защитить. Плюс два ⚠️: guard против «billion laughs» в XML **обходится** одним символом `>` в SYSTEM-литерале, и **токен JWT может транзитом попасть в `storage.session`** через handoff выделения, что противоречит 🔴-инварианту дизайна «токен — только RAM».

## (b) 🔴 БЛОКЕРЫ

### B1. Автодетект формата парсит весь документ `JSON.parse` на главном потоке — воркер-архитектура обойдена на дефолтном пути
`utils/core/detect.ts:137-143` (внутри `detectFormat`, ветка `{`/`[`):
```ts
if (hasJsonComments(text)) return { format: 'jsonc', confident: true };
try {
  JSON.parse(text);                       // ← весь документ, на главном потоке
  return { format: 'json', confident: true };
} catch { return { format: 'json5', confident: false }; }
```
Путь вызова целиком на главном потоке: `useDocument.load()` → `resolveFormat(text, pref)` (`utils/document.ts:153-154`) → `format.ts:35-42` → `detect.ts:124`. `resolveFormat` вызывает `detectFormat` всегда, когда `pref === 'auto'`, а `DEFAULT_PREFS.defaultFormat = 'auto'` (`utils/storage.ts:76`). То есть для **любой** авто-детектируемой вставки/дропа JSON:

1. **Фриз до появления UI.** `resolveFormat` отрабатывает **синхронно и до** `run()`, который единственный переводит состояние в `phase:'loading'` и постит задачу в воркер (`document.ts:154-164`, `:80-88`). Значит `JSON.parse` блокирует главный поток **раньше**, чем нарисуется индикатор и кнопка «Отменить». Кнопка «Отменить» (дизайн §5.1, обязательная) в этот момент физически не может помочь — поток занят не в воркере.
2. **Выброшенный граф → риск OOM.** Результат `JSON.parse` **не используется** — он нужен только чтобы отличить `json` от `json5`. Для документа у `HARD_MAX_BYTES = 200_000_000` (`format.ts:29`) это ~600 МБ–1 ГБ временного графа объектов на главном потоке, который тут же выбрасывается, — до того как воркер (который должен был держать этот документ) вообще стартует.
3. **`hasJsonComments` тоже полный проход по строке** на главном потоке (`detect.ts:66-84`, посимвольно), плюс `looksLikeJwt` делает `text.split('.')` на всём тексте (`detect.ts:35`).

**CONFIRMED** (node, V8, результат отбрасывается как в коде): `JSON.parse` 20 МБ = **120 мс**, 50 МБ = **333 мс**. 50 МБ — это ровно заявленная в дизайне (§0, §1.2) и в `TODO.md` §F цель «**50 МБ без фриза**, детект формата **в Worker**». Дизайн §4.1 прямо пишет «Текст → **Worker**: детект формата (BOM → `{`/`[` → …) → парс». Реализация делает детект (и полный парс ради детекта) на главном потоке. Комментарий `detect.ts:8-9` («runs identically on the main thread and in the Worker») формально верен, но по факту функция вызывается **только** на главном потоке, и делает она не дешёвую эвристику, а полный `JSON.parse`.

**Почему блокер, а не «важное»:** это тихий обход центральной архитектурной гарантии продукта — на дефолтной настройке, для целевого пользователя (разработчик с большим JSON), — который к тому же способен уронить вкладку по памяти до старта воркера. Заявленная киллер-фича «50 МБ без фриза» на авто-детекте не работает.

**Фикс:** детект не должен полностью парсить. Отличать `json`/`json5` дешёвой эвристикой (пробное `JSON.parse` только на первых N КБ, либо вообще отложить различение до воркера и парсить как `json` с фолбэком на `json5` при ошибке **в воркере**). Идеально — перенести `detectFormat` в воркер, как и написано в дизайне §4.1, и на главном потоке держать только выбор «это точно не текст → показать спиннер». `hasJsonComments`/`looksLikeJwt` над мегабайтами — туда же.

> ✅ **ИСПРАВЛЕНО 2026-07-14.** `detectFormat` больше не парсит: brace-ветка по структуре считает документ строгим JSON, `hasJsonComments` ограничен первыми 256 КБ, `looksLikeJwt` ищет две точки через `indexOf` (не `split('.')`), `firstLineOf` заменил `split('\n')[0]`. Решение JSON→JSONC→JSON5 ушло в воркер (`parseText` ретраит off-thread, фактический формат возвращается через `ParseSuccess.format`). Замерено: детект 55 МБ = **31 мс, 0 полных парсов** (было ~333 мс + ~1 ГБ мусорного графа). Валидный JSON остаётся `json` → точные числа целы.

## (c) ⚠️ ВАЖНОЕ

### V1. Guard против «billion laughs» в XML обходится символом `>` в SYSTEM-литерале
`utils/core/xml.ts:29`:
```ts
const DTD_ENTITY = /<!DOCTYPE[^>]*\[[\s\S]*?<!ENTITY/i;
```
Комментарий `xml.ts:9-11` обещает: «A DTD carrying entity definitions is **REFUSED outright**». Это не так. `[^>]*` не может пересечь `>`, а `>` **допустим внутри SystemLiteral** (`SystemLiteral` = любой символ кроме кавычки). Поэтому документ с `>` в SYSTEM-литерале **до** внутреннего подмножества `[…]` проходит мимо guard'а:

**CONFIRMED** (node): 
- `<!DOCTYPE r [<!ENTITY a "x">]>…` → guard срабатывает (`true`). 
- `<!DOCTYPE r SYSTEM "a>b" [<!ENTITY lol "…"><!ENTITY lol2 "&lol;&lol;">]>…` → guard **не** срабатывает (`false`), документ уходит в `DOMParser` с объявленными сущностями.

XML разбирается на **главном потоке** (`format.ts:65-66`, `:107-137`: `DOMParser` в воркере нет), под лимитом `MAX_XML_BYTES = 20_000_000` (`xml.ts:22`). Лимит на размер **входа** не защищает от экспоненциальной развёртки сущностей — суть billion-laughs в том, что маленький вход даёт огромную память. **PLAUSIBLE, не CONFIRMED:** реальный отказ по памяти зависит от собственных лимитов раскрытия сущностей у `DOMParser` браузера (Chrome/Firefox их ограничивают, и при превышении дают `parsererror`, который `xml.ts:47-51` ловит в `XmlRefused`). Но guard, на который явно ссылается дизайн (§4.6) и комментарий, **дырявый**, и XML-путь всё равно синхронный на главном потоке.

**Фикс:** не полагаться на regex по DTD. Достаточно `text.slice(0, N)` и проверки наличия `<!ENTITY` где угодно во входе (сущность вне DTD не бывает валидной, а лишний отказ здесь безвреден), либо `parseFromString` с последующей проверкой, что документ не содержал `<!DOCTYPE …[…]>` через `doc.doctype`. И честно: раз «мы отказываемся играть», отказывать по факту любого `<!ENTITY`/непустого `doctype.internalSubset`, а не по хрупкому шаблону.

> ✅ **ИСПРАВЛЕНО 2026-07-14.** Хрупкий regex заменён на `declaresEntities()` — сканер DOCTYPE, пропускающий `'…'`/`"…"`-литералы и отслеживающий глубину `[…]`, закрывающий DOCTYPE только на `>` вне обоих. Отбивает подтверждённый обход (`<!DOCTYPE r SYSTEM "a>b" [<!ENTITY …]>`) и одинарно-кавычечный вариант, при этом `<!DOCTYPE html>` пропускается. Комментарий приведён в соответствие.

### V2. Токен JWT может транзитом попасть в `storage.session` через handoff выделения — против 🔴-инварианта «токен только RAM»
`entrypoints/background.ts:93` → `utils/handoff.ts:27` (`putHandoff`) → `handoff.ts:23` (`storage.defineItem('session:handoff')`).
Контекстное меню «Открыть выделенное в Data Toolkit» кладёт `info.selectionText` в `storage.session`. Если пользователь выделил на странице **сам JWT** (типичный сценарий: токен в логе/ответе API) и нажал пункт меню — токен оказывается в `session:handoff` до момента, пока tool page его не прочитает и не удалит (`takeHandoff`, `document.ts:206-210`, там он детектится как JWT и предлагается табу JWT).

Это противоречит:
- дизайну §3 (инвентарь): «**Токен JWT — RAM. Не персистится ни при каких настройках.**» (🔴);
- собственному комментарию `handoff.ts:11-12`: «The JWT tab NEVER writes here. **Nothing that could carry a credential is handed off or persisted** anywhere in this extension.» — это утверждение ложно для канала выделения/буфера.

**Blast radius низкий и это честно надо сказать:** `storage.session` — память, не диск, чистится при закрытии браузера, читается один раз и сразу удаляется (`handoff.ts:42-46`), плюс дроп по возрасту >60 с. Но `storage.session` — это **extension storage**, переживающий смерть service worker и читаемый из любого контекста расширения; это не «RAM вкладки», как обещает инвентарь. **Секрет HS256 и ключи сюда не попадают** (они только в `JwtTab` state) — проверено. Проблема именно в том, что абсолютный инвариант и явный комментарий сформулированы шире, чем правда.

> ✅ **ИСПРАВЛЕНО 2026-07-14.** `putHandoff` детектит `looksLikeJwt` и **отказывается класть** такой текст в session, возвращая `jwt-skipped` (новый `HandoffOutcome`). Вызыватели (`background.ts` — контекстное меню, `popup/App.tsx` — буфер) на `jwt-skipped` открывают JWT-таб, куда токен вставляется в RAM-only стейт компонента, минуя `storage.session`. Инвариант «токен только RAM» снова верен буквально; переширокий комментарий исправлен.

**Фикс:** либо в `putHandoff` детектить `looksLikeJwt(text)` и **не** класть такой текст в session (передавать флаг «был JWT» без тела, tool page сам предложит вставить вручную), либо переписать инвариант/комментарий честно: «токен может кратко пройти через session-handoff, если пользователь сам его выделил; на диск не пишется никогда».

## (d) 💡 УЛУЧШЕНИЯ

- **`sync:prefs` — допустимо, но на грани.** `utils/storage.ts:88` кладёт `DevdataPrefs` в `storage.sync`. Дизайн §0/§3 это санкционирует (объект ~300 Б, кап 8 КБ/элемент не грозит), и это осознанно отличается от `blur`/`compose`, где всё в `local`. ОК, но стоит держать `DevdataPrefs` плоским и маленьким навсегда — любое поле-массив (история путей, недавние документы) сюда добавлять нельзя.
- **XML на главном потоке (20 МБ).** Даже с закрытым V1 `DOMParser` 20 МБ — это заметная синхронная работа на главном потоке (`format.ts:116`). Дизайн это признаёт (§10.4), лимит есть, но стоит спрофилировать и, возможно, снизить `MAX_XML_BYTES`, либо резать разбор на чанки/`requestIdleCallback` при рендере дерева.
- **`stripKeyword` рекурсивен по схеме** (`data.worker.ts:318-338`) — глубокая враждебная схема может переполнить стек; поймано `try/catch` и это в воркере, так что не опасно, но `externalRefs` рядом сделан итеративно — стоит и `stripKeyword` сделать итеративным для единообразия.
- **`convertSubtree` копирует поддерево на главном потоке** (`format.ts:239-247`, `.slice().map()`), затем шлёт в воркер. Для большого поддерева копия строится на главном потоке — мелко, но при 50 МБ-документе может дать джанк; можно передавать индексы и резать в воркере.

## (e) ✅ Проверено и в порядке (не переделывать)

- **Сети нет вообще.** 0 `fetch`/`XMLHttpRequest`/`WebSocket`/`sendBeacon`/`EventSource` во всём `extensions/devdata` (grep). `jose`, `@cfworker/json-schema`, `yaml`, `json5`, `papaparse` подключаются **динамическим `import()`** и бандлятся Vite локально — ни одного удалённого URL, никакого JWKS-fetch (в `jwt.ts` его физически нет). `data_collection_permissions.required = ['none']` в Firefox-манифесте — верно.
- **Секреты не утекают.** HS256-секрет, приватный и публичный ключ существуют **только** в state `JwtTab` (`JwtTab.tsx:23-25`) и нигде не пишутся в storage/draft/history — соответствующего storage-item просто нет (`storage.ts:18` фиксирует это как инвариант). Поле секрета: `type=password`, `autocomplete=off`, `spellcheck=false`, `data-1p-ignore`, reveal «нажать-и-держать», не sticky (`JwtTab.tsx:189-214`). Ошибки верификации не эхоят материал ключа/секрета (`jwt.ts:272-302`; приватный ключ детектится по `d`/`BEGIN PRIVATE KEY` и отклоняется без вывода тела). `console.*` — ноль.
- **DoS/зависания заперты в воркере.** `runJob` создаёт **одноразовый** воркер на задачу, `dispose()` реально `worker.terminate()` по результату/ошибке/`messageerror`/таймауту/`cancel` (`worker/client.ts:84-147`); `cancel` **реджектит** (не висящий промис); UI восстанавливается (`document.ts:82-121` разбирает `timeout`/`worker-died`/`cancelled`). ReDoS в схеме заперт: `pattern` исполняется только внутри воркера, 5 с → terminate (`data.worker.ts:251-289`, `client.ts:42`). Внешние `$ref` — явный отказ, не тихий пропуск (`data.worker.ts:226-268`). Итеративные обходы дерева/сущностей (`xml.ts:80-129`, `data.worker.ts:229-248`) — без рекурсии по глубине пользователя.
- **In-page форматтер безопасен.** `formatter.content.ts`: только `document.createElement` + `textContent`, ни одного `innerHTML` (`:86`, `:171`, `:276`); дерево строится **итеративно** с кап-глубиной 100 (`:233-281`), кап размера 4 МБ (`:26`, `:192`); one-shot vs auto разведены через запрос к фону (`:110-124`); `✕` возвращает сохранённый оригинал (`:83-89`). `registration:'runtime'` держит скрипт вне манифеста → ноль host-предупреждений на инсталле.
- **Разрешения и single purpose.** `permissions: storage, contextMenus, activeTab` (0 предупреждений), `scripting` + `<all_urls>` — optional и **реально используются** (`permissions.ts`, `format-page.ts`: `executeScript`/`registerContentScripts`, MV2-фолбэк `tabs.executeScript`/`contentScripts.register`). Permission-**факт** читается из `permissions.contains` и живо обновляется на `onAdded`/`onRemoved` (`permissions.ts:78-101`) — оптимистичного тумблера-лжи (баг adblock из §18a) нет. Firefox MV2 честно трактует отсутствие `scripting` (`permissions.ts:27-34`). `options_ui → tool.html#/settings, open_in_tab: true`. Ни одной второй иконки/omnibox/devtools-панели (дизайн §1.4 соблюдён).
- **Storage-модель.** Всё крупное — `local:` (`document` ≤1 МБ, `schema` ≤256 КБ) с явными кап-проверками и **видимыми** отказами (`skipped-too-big`/`failed`, `storage.ts:149-181`, `document.ts:262-277`) — тихой лжи «saved» из `blur`/`sync` нет. `version` + `migrations` объявлены у всех items.
- **Мобильный (Firefox Android).** Главная поверхность — полная вкладка `tool.html`, открывается из popup и контекстного меню через `tabs.create` (`background.ts:28-35`) — всё это на Android есть. Деградация штатная (дизайн §1.2.5), мёртвых фич нет.
- **Манифесты.** Нет `web_accessible_resources`, нет переопределения CSP сверх дефолта, нет remote code, нет `debugger`/`webRequest`. `gecko.id` на реальном домене, `gecko_android: {}` присутствует. Single purpose держится.

## (f) Что стоит покрыть тестами (сейчас дыры видны глазами)

1. **B1** — тест «авто-детект большого JSON не блокирует главный поток дольше X мс» / «детект не делает полный `JSON.parse`». Сейчас регресс невидим.
2. **V1** — XML с `>` в SYSTEM-литерале и объявленными сущностями: тест, что guard **отказывает** (сейчас пропускает).
3. **V2** — выделение-JWT через контекстное меню: тест, что токен **не** оказывается в `session:handoff` (или что инвариант переписан).
4. **Секреты** — тест, что после верификации HS256 `storage.session`/`storage.local`/`storage.sync` не содержат ни секрета, ни токена (регресс-страховка на будущие правки handoff).
5. **Воркер-таймаут схемы** — тест с ReDoS-`pattern`, что через 5 с приходит `timeout` и UI живой (сейчас поведение заявлено, но не закреплено).
6. **Точные большие числа** — что `12345678901234567890` показывается как есть, а не округляется (киллер-фича, `format.ts:305-309`), и что при недоступности source access UI честно помечает «недоступно».
