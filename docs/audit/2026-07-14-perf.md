# Аудит `extensions/perf` (Page Performance & Network) — 2026-07-14

Аудитор: senior code auditor. READ-ONLY по исходникам. Версия 1.0.0, MV3, Chrome + Firefox.
Фокус — приватность (единственное расширение с сетевым вызовом: PageSpeed Insights) и CWS-политика от 2026-08-01.

---

## (a) Вердикт

Инженерно расширение зрелое: честность байтов (`transferSize=0` vs `null`), opt-in `debugger` с объяснением баннера до `permissions.request()`, гарантированный `detach` в `finally`, Firefox-гарды, все регрессии §18c проверены и держатся. Но по приватности PSI есть **реальный блокер, которого нет в TODO**: в UI Google уходит **полный URL страницы вместе с query-string** (`?token=…`, `?reset=…`, PII), при этом (1) поле URL в панели **нередактируемо** — пользователь не может вычистить секрет, (2) раскрытие об этом **молчит**. Вопреки посылке задания и TODO, in-UI-раскрытие PSI **уже существует** (гейт с affirmative consent до первого вызова в `AuditPanel.tsx`), т.е. голое требование политики «prominent + in-UI + до сбора» формально выполнено — но раскрытие тонкое и с дырами (нет data-retention, нет ссылки на политику, нет предупреждения о секретах в URL, нет revoke). Итог: не блокирует по «отсутствию раскрытия», блокирует по «утечке секретов из URL без предупреждения и без возможности отредактировать».

## (b) 🔴 БЛОКЕРЫ (до сабмита)

### B1. Полный URL с секретами уходит в Google, без предупреждения и без возможности редактирования
`extensions/perf/utils/psi.ts:204` (`endpoint.searchParams.set('url', url)`) + `extensions/perf/entrypoints/panel/AuditPanel.tsx:34-38, 106`.

- URL берётся из `browser.devtools.inspectedWindow.eval('location.href')` — это **полный** href с query и fragment усечённым браузером, но query остаётся. `runPsiAudit` отправляет его как есть в `www.googleapis.com`.
- Поле показа URL — `<div className="audit-url">` (AuditPanel.tsx:106), **read-only**. Пользователь физически не может убрать `?token=SECRET` перед отправкой. PLAN §14 (стр. 106) прямым текстом: «PSI API отправляет URL Google — это надо явно раскрыть», а PLAN §9 упоминает «user-entered page URL» — но ввод/правка URL в UI отсутствуют.
- Раскрытие (см. B2) говорит только «page URL will be sent to Google» — ни слова о том, что вместе с ним уходят query-параметры (сессионные токены, ссылки на сброс пароля, поисковые запросы, email в URL).

Почему это блокер: privacy-first набор, инвариант «данные о страницах не уходят наружу», а тут единственный сетевой путь молча эксфильтрует потенциальные секреты стороне (Google), причём пользователь лишён средства это предотвратить. Под политику 2026-08-01 (сбор должен быть *strictly necessary* и prominently disclosed) отправка **сырых query-строк** не является strictly necessary и не раскрыта.

Фикс (конкретно):
1. Сделать URL **редактируемым** `<input>` (значение из `inspectedWindow.eval` как дефолт), чтобы пользователь мог вычистить query перед аудитом.
2. Показать распарсенные query-параметры, если они есть, с предупреждением и кнопкой «Убрать параметры» (audit по `origin+pathname`). PSI по origin/pathname обычно достаточно для лабораторного прогона.
3. Расширить текст раскрытия (B2), явно назвав query-параметры.

### B2. In-UI раскрытие PSI не дотягивает до prominent disclosure по политике 2026-08-01
`extensions/perf/entrypoints/panel/AuditPanel.tsx:126-137` (алертдиалог) + `83-89` (note).

Состояние (проверено в коде, подтверждает claim параллельного ревью whoami):
- Гейт-диалог `role="alertdialog"` содержит **одно предложение**: «To audit, you must acknowledge that the page URL will be sent to Google's PageSpeed Insights API.»
- Нет: заявления о **хранении/удержании** данных Google, ссылки на политику Google/на `PRIVACY.md`, упоминания query-параметров/секретов (см. B1), пути **отзыва** согласия.
- `disclosureAccepted` пишется в `local:psiConfig` один раз (`AuditPanel.tsx:50-53`) и **никогда** не сбрасывается — UI отзыва нет (проверено: `permissions.remove`/reset отсутствуют во всём `entrypoints/`). Согласие «навсегда».
- Раскрытие живёт только в **DevTools-панели**. Это ОК (PSI только там), но панель проверяется лишь своими e2e — риск, что реальный ревьюер CWS его не увидит без открытых DevTools (см. раздел f).

Почему блокер (пограничный, но до сабмита закрыть): политика требует *prominent* раскрытия и уведомления об изменении практик. Односентенсовый диалог без retention и без revoke — минимально-формальное, «prominent» спорно; в связке с B1 (не сказано про секреты) не проходит.

Точная копия UI, удовлетворяющая политике (заменить тело `confirm` и добавить revoke):

> **Этот аудит отправляет URL в Google**
> Чтобы прогнать PageSpeed Insights, расширение отправит **полный адрес этой страницы, включая параметры запроса после «?»**, в PageSpeed Insights API Google (`www.googleapis.com`). Google загрузит и измерит страницу и вернёт результаты.
> ⚠️ Параметры запроса могут содержать приватные данные — токены сессии, ссылки для сброса пароля, поисковые запросы. **Проверьте и при необходимости отредактируйте адрес выше перед запуском.** Убрать все параметры одной кнопкой: «Только домен и путь».
> Отправляется только при явном запуске аудита. Переданный Google адрес подчиняется [политике конфиденциальности Google]. Полностью локальные метрики (Web Vitals, resource timing, exact-bytes) не отправляют ничего.
> [Отмена] [Понимаю — отправить адрес в Google]
>
> (после согласия) мелкой строкой в разделе Audit: «PSI-раскрытие принято. [Отозвать согласие]» → сбрасывает `disclosureAccepted=false`.

Плюс: в `PRIVACY.md:225-226` уточнить, что уходит **URL целиком с query**, а не «the URL being audited».

## (c) ⚠️ ВАЖНОЕ

### C1. Same-page-скрипт может подделать метрики (осознанный остаточный риск, но это «вранье в UI»)
`extensions/perf/entrypoints/content.ts:31-39`, `utils/protocol.ts:53-71`, `relay.content.ts:40-51`.
Nonce хранится в атрибуте `data-blur-perf-nonce` на `documentElement` — читаем любым скриптом страницы. Cross-frame-подделка закрыта (проверка `event.source`, `event.origin`, nonce — верно), но **same-page** скрипт может прочитать nonce и запостить фейковые LCP/CLS/LoAF, которые расширение покажет как реальные. Задокументировано в `protocol.ts:56-62` как принятый компромисс (вынос сбора из MAIN world требует core-изменений). Impact: нарушение house-rule «не врать в UI» на враждебной странице; эскалации привилегий нет. Для прода — не блокер, но зафиксировать в известных ограничениях и не показывать метрики как «доверенные».

### C2. Смерть SW во время CDP-замера оставляет debugger приаттаченным + жёлтый баннер
`extensions/perf/utils/debugger-bytes.ts:262-322`.
`detach()` в `finally` закрывает нормальный путь и путь ошибки. Но между `Page.reload` и гонкой 20с+1.5с висит цепочка `await`; если MV3 SW будет вытеснен в этом окне, `finally` не выполнится — debugger останется attached с несбрасываемым баннером до ручного вмешательства/закрытия вкладки. MV3 не гарантирует живучесть SW под открытым CDP-соединением (обычно attached debugger держит SW, но не гарантированно на 20с). Митигировать: (1) слушать `chrome.debugger.onDetach` для очистки состояния (уже есть listener, но он только резолвит load); (2) добавить на старте `background` sweep `chrome.debugger.getTargets()` → detach «наших» повисших; либо документировать как редкий крайний случай.

### C3. `debugger` остаётся в granted optional-permissions после замера
`extensions/perf/entrypoints/popup/App.tsx:100`.
После `permissions.request(['debugger'])` разрешение остаётся выданным (детачится только CDP-сессия, не разрешение). Для минимизации поверхности и «Purple Potassium» — после замера не критично (баннер уходит при detach), но чище звать `permissions.remove({permissions:['debugger']})` по завершении/по кнопке. Не блокер.

### C4. PSI-таймаут 60с, но кнопка/UX
`utils/psi.ts:43` `PSI_TIMEOUT_MS=60_000`, комментарий на строке 40 верен. AbortController есть (bug 1i закрыт). ОК по сути; убедиться, что 60с согласованы с ожиданиями (лаб-прогон PSI бывает медленным — это ОК).

## (d) 💡 УЛУЧШЕНИЯ

- **D1.** `AuditPanel.tsx:35` `inspectedWindow.eval('location.href')` — на SPA после client-side навигации href может устареть (эффект без зависимостей, читается один раз). Перечитывать по `chrome.devtools.network.onNavigated` или при открытии таба Audit.
- **D2.** `psi.ts:207` API-ключ уходит в query-string (`key=`) — стандарт для PSI, но ключ попадёт в возможные логи прокси. Приемлемо (так требует Google), отметить в комментарии.
- **D3.** `registrable-domain.ts` — ручной список multi-label суффиксов; помечает только *флаг* third-party, не байты (задокументировано, ОК). При росте — рассмотреть компактный PSL. Не срочно.
- **D4.** `web-vitals ^5.3.0` с `web-vitals/attribution` — актуально на 2026-07, attribution-сборка используется (`content.ts:2`), все `onX` вызываются по разу под guard-флагом (`content.ts:69`). LoAF + Long Tasks fallback — современно. Замечаний по свежести стека нет. TS `^7.0.2`, React `^19`, WXT `^0.20.27` — консистентно.
- **D5.** Popup опрашивает background `setInterval(1500)` (`popup/App.tsx:87`) и панель тоже (`panel/App.tsx:127,258`) — polling вместо push. Работает, но лишние сообщения; не блокер.

## (e) ✅ Проверено и в порядке

- **Честность байтов.** `resolveTransferSize` (`resource-timing.ts:60-64`) не путает cache-hit `0` (при `decodedBodySize>0`) с неизмеримым `null`. `transferSize=null` → «unmeasured», нигде не 0. Проверено e2e (`perf.spec.ts:97,349`).
- **`ignoreCache:true`** добавлен (`debugger-bytes.ts:280`) — тёплый кэш больше не даёт `encodedDataLength≈0`; метка «exact bytes (cold load)» (`popup/App.tsx:217`) честная.
- **«Unmeasured» для обоих источников** (bug 1k): RT и HAR (`panel/App.tsx:317,352`). PSI null-score → «—», не «0» (`psi.ts:185`, `AuditPanel.tsx:150`).
- **Compare не мешает источники** (bug 1d/§18c): `bytesComparable` требует одинакового `byteSource` (`popup/App.tsx:197,352`); cross-host не сравнивается.
- **§18c регрессии — все держатся:** `getState` перепроверяет live-кэш после await (`background.ts:131`); `forgotten`-guard против воскрешения LCP (`background.ts:57,135`); `stop()` только на терминальном unload + рестарт на `pageshow{persisted}` (`content.ts:168-173`, `relay.content.ts:91-96`, коллекторы идемпотентны через `started`); HAR берётся только если не короче RT (`panel/App.tsx:301`); `persist()` debounce 600мс (`background.ts:62,86-106`); prune stale `perf:tab:*` только на local-fallback (`background.ts:157-179`).
- **CDP load-event гонка** (bug 1b): `loadPromise` армится до `Page.reload` (`debugger-bytes.ts:271-280`).
- **`debugger` opt-in корректен:** запрос по user-gesture (`popup/App.tsx:100`), баннер объяснён в `ConsentDialog` **до** запроса (`popup/App.tsx:270-271, 418-442`), detach в `finally` (`debugger-bytes.ts:321`), Firefox-гард в коде (`debugger-bytes.ts:195`) и в манифесте (`optional_permissions:['debugger']` только не-Firefox, `wxt.config.ts:65-70`). UI-кнопка скрыта на Firefox (`popup/App.tsx:256`).
- **AMO `data_collection_permissions`** (`wxt.config.ts:98-101`): `required:['none']`, `optional:['websiteActivity']` — консистентно с кодом (по умолчанию сеть не трогается; URL страницы = websiteActivity, за opt-in host-permission). `technicalAndInteraction` не в required (его там и нет). Соответствует `PRIVACY.md:96,225` и `STORE.md:56`. Верно.
- **Безопасность:** нет `innerHTML`/`eval`/`new Function`/`dangerouslySetInnerHTML` (grep чист; два `inspectedWindow.eval` — это DevTools-API чтения `location.*`, не JS-eval недоверенного). Валидация cross-world сообщений корректна (`relay.content.ts:41-43`). `web_accessible_resources` не объявляются.
- **Метрики без фейков:** grep на mock/48907/fabricated/placeholder — чисто (только строковые комментарии и `placeholder="AIza…"` в input). LoAF/LongTask деградируют в «neither exposes…» (`LongFrames.tsx:68-74`), не в 0.
- **A11y/тема:** панель — `data-theme` от `themeName` + live `onThemeChanged` (`panel/main.tsx:11-25`), `color-scheme` под тему DevTools (`panel/style.css:15,47,67,94`), `focus-visible` (обе поверхности), tab/tabpanel с клавиатурной навигацией стрелками/Home/End (`panel/App.tsx:50-63`), `<caption>`/`scope`/`aria-live`, waterfall с видимой легендой (не только цвет, `Waterfall.tsx:174-181`). Severity vitals — текстом (`ratingLabel`), не только цветом.
