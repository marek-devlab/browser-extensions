# Markdown Workbench (`extensions/compose`) — UX/UI дизайн

> **Версия 1** (2026-07-14). Дизайн-документ расширения №10 из [`PLAN.md`](../../PLAN.md) (Часть II) §6, §10.6.
> Цель одной фразой (не подлежит расширению): **«Написать и отформатировать текст перед вставкой»**.
> Разрешения: `storage`, `contextMenus`, `clipboardWrite`, `activeTab`. **Ноль install-предупреждений. Ноль сети. Ноль удалённого кода.**
>
> ⚠️ — места, где «очевидное» решение ломает продукт. 🔴 — нарушение = отказ ревью или дыра в безопасности.
> Конвенции дома: `PLAN.md` §18a–18d (тема через `data-theme` + `color-scheme`, `aria-live`, «не врать в UI», **`storage.local`, не `sync`**).

## 1. Информационная архитектура

### 1.1 🔴 Главный инвариант: у расширения ОДНА точка входа

Проверка ревьюера — «помещается ли цель в одну фразу». Наша фраза — «написать и отформатировать текст перед вставкой». Regex, транслит и счётчик проходят **только как операции над черновиком, который открыт прямо сейчас**. Формальное правило, которое обязано соблюдаться в коде и в манифесте:

| Инвариант | Как проверяется механически |
|---|---|
| В манифесте **один** `action` | `action` открывает side panel. Больше UI-иконок нет |
| В манифесте **нет** `commands`, открывающих отдельный инструмент | Есть ровно один `_execute_action`-подобный шорткат «открыть черновик». Остальные шорткаты работают **внутри** редактора |
| В `contextMenus` — **один** пункт | «Добавить выделенное в черновик». Не «транслитерировать выделенное», не «посчитать символы» |
| Ни один инструмент не рендерится без загруженного черновика | Панели Find&Replace / Translit / Stats — дочерние компоненты `<Editor>`, монтируются только внутри него |
| В листинге и в `description` — **ни разу** слова «regex tester», «character counter», «transliterator» как самостоятельных продуктов | Они перечислены как «внутри редактора: …» |

🔴 **Момент, где расширение умирает:** отдельный popup «Regex», отдельная иконка, отдельный пункт контекстного меню «Транслитерировать выделенное на странице» (это уже операция над **страницей**, а не над черновиком → другой продукт). Соблазн большой, цена — Red-семейство кодов отказа.

⚠️ Обратная сторона того же правила: контекстное меню **не изменяет страницу**. Оно только **вливает** выделение в черновик. `activeTab` используется на чтение (`window.getSelection()` через `scripting.executeScript` по жесту) и никогда на запись.

### 1.2 Поверхности

| # | Поверхность | Файл | Роль | Ширина |
|---|---|---|---|---|
| **S1** | **Side panel** (Chrome `side_panel`) / **Sidebar** (Firefox `sidebar_action`) | `entrypoints/sidepanel/` | 🟢 **Основная.** Пишешь баг-репорт, глядя на страницу | 320–600 px, resizable пользователем |
| **S2** | **Workbench** — полноэкранная страница расширения | `entrypoints/workbench/` | Длинные документы, широкий split-view, история черновиков | 700–1600+ px |
| **S3** | **Options** | `entrypoints/options/` | Только настройки: тема, дефолтная площадка, стандарт транслита, шаблоны, лимиты, таймаут воркера | 640 px колонка |
| **S4** | **Context menu** | background | `Добавить выделенное в черновик` (+ подпункт `…как цитату`) | — |
| **S5** | Toolbar action | background | Открывает S1. **Не popup.** | — |

⚠️ **S1 и S2 — это ОДИН React-компонент `<Workbench>`**, отличающийся только контейнером и брейкпойнтом (`container-type: inline-size` + `@container`). Никакого второго редактора, никакой второй раскладки в коде. `S2` — это `S1`, которому дали 1400 px и который поэтому показывает split-view вместо табов.

⚠️ **`side_panel` vs `sidebar_action` — открытый вопрос §11 PLAN.md (Часть II).** Дизайн обязан работать в обоих, поэтому:
- Не полагаемся на `sidePanel.setOptions({ tabId })` (per-tab панели) — в Firefox аналога нет. Панель **глобальная**, черновик один и тот же во всех вкладках. Это ещё и правильнее продуктово: черновик не должен исчезать при переходе на другую вкладку.
- Не полагаемся на программное открытие панели из фона без жеста. Chrome: `sidePanel.open()` требует user gesture; Firefox: `sidebarAction.open()` — тоже. Значит **контекстное меню считается жестом** и панель откроется; а вот «автоматически открыть панель, когда пользователь что-то скопировал» — 🔴 невозможно и не проектируем.
- Firefox: `sidebar_action.default_panel` → тот же HTML. Кнопка «Открыть в отдельной вкладке» (S2) есть в обоих — это страховка, если панель на конкретной сборке ведёт себя плохо.

### 1.3 Как regex / транслит / счётчик вложены в редактор

```
<Workbench>
├── Header: [черновик ▾] [площадка ▾] [⟲ история] [⛶ вкладка] [тема]
├── EditorPane            ← ЕДИНСТВЕННЫЙ источник истины: markdown-текст
│   ├── Toolbar           ← B / I / code / list / [ ] / <details> / table / link / 😊
│   └── <textarea>        ← плоский текст + оверлей подсветки (Custom Highlight API)
├── PreviewPane           ← только чтение, санитизированный DOM
├── ToolDrawer            ← ОДИН выдвижной ящик, три вкладки, всегда над черновиком
│   ├── [Найти и заменить]   ← regex по ЭТОМУ черновику
│   ├── [Транслитерация]     ← по выделению В ЭТОМ черновике
│   └── [Статистика]         ← числа ЭТОГО черновика + лимиты площадки
└── CounterStrip          ← всегда виден, одна строка, клик → разворачивает вкладку «Статистика»
```

**Ключевые ограничения, делающие их «частями редактора», а не инструментами:**
- ToolDrawer **не имеет собственного URL/entrypoint**. Его нельзя открыть, не открыв редактор.
- Все три панели работают **только** с текстом текущего черновика. Нет поля «вставьте сюда текст для проверки regex» — поле ввода — это сам редактор. Это ровно то, что отличает нас от regex101 в обёртке.
- Кнопка «Применить» в каждой панели **пишет обратно в черновик** (undo-able). Панель, которая только показывает результат и не может его вернуть в текст, — признак того, что вы построили отдельный инструмент.
- Счётчик не умеет считать «произвольный текст» — только черновик.

### 1.4 Модель данных

| Сущность | Хранение | Почему |
|---|---|---|
| `drafts: Draft[]` (id, title, body, target, createdAt, updatedAt) | `local:drafts` | ⚠️ **Никогда `sync`.** Квота `sync` — 8 192 байта **на элемент**, длинный баг-репорт её порвёт. На этом уже обожглись в `blur` (`PLAN.md` §18a) |
| `activeDraftId` | `local:activeDraftId` | |
| `history: Snapshot[]` (кольцевой буфер, до 30 на черновик) | `local:history` | Автоснимки + именованные точки |
| `settings` | `local:settings` | Мелкий объект, но кладём в `local` из принципа дома: никаких квот-сюрпризов |
| `unsavedBuffer` | `session:unsaved` | Черновик между автосейвами; переживает смерть service worker'а, но не перезапуск браузера |
| Эмодзи-данные | лениво `await import()` | ⚠️ Не в основном бандле (§10.3) |

### 2.3 Toolbar — что именно вставляет каждая кнопка

```
┌─────────────────────────────────────────────────────────────────────────┐
│ B      **жирный**                    Ctrl+B                             │
│ I      *курсив*                      Ctrl+I                             │
│ S̶      ~~зачёркнутый~~               Ctrl+Shift+X                       │
│ <>     `код` / ```блок```            Ctrl+E / Ctrl+Shift+E              │
│ •      - пункт                       Ctrl+Shift+8                       │
│ 1.     1. пункт                      Ctrl+Shift+7                       │
│ ☑      - [ ] задача                  Ctrl+Shift+C   ← ЗАПРОШЕНО          │
│ ⌄      <details><summary>…           Ctrl+Shift+D   ← ЗАПРОШЕНО          │
│ ▦      таблица 3×2 (диалог)          Ctrl+Shift+T                       │
│ 🔗     [текст](url)                  Ctrl+K                             │
│ "      > цитата                      Ctrl+Shift+.                       │
│ 😊     эмодзи-пикер (popover)        Ctrl+Shift+J                       │
└─────────────────────────────────────────────────────────────────────────┘
```

⚠️ На узкой панели toolbar **не скроллится горизонтально** (это ад для тач- и клавиатурной навигации) — лишние кнопки уезжают в `⋯` (меню-popover) в порядке приоритета: B, ☑, `<>`, ⌄, 😊 остаются всегда.

Вставка `<details>` — не голый тег, а рабочая заготовка (⚠️ **пустые строки вокруг тела обязательны**, иначе GitHub не парсит Markdown внутри):

```
<details>
<summary>КАРЕТКА</summary>

тело

</details>
```

### 2.4 Эмодзи-пикер — Popover API, не модалка

```
        😊  ← кнопка toolbar (popovertarget)
        ┌────────────────────────────────────────┐
        │ [🔍 поиск: rocket▍            ]  [×]   │
        │ ────────────────────────────────────── │
        │ Недавние                               │
        │  🚀  🐛  ✅  ⚠️  🎉  👍  🔥  📌         │
        │ ────────────────────────────────────── │
        │ Результаты «rocket»                    │
        │  🚀 :rocket:                           │
        │  🛸 :flying_saucer:                    │
        │ ────────────────────────────────────── │
        │ Вставить как:                          │
        │  ( ) Символ 🚀    (•) Шорткод :rocket:  │
        │  ⚠️ Jira и Telegram шорткоды не        │
        │     понимают — для них вставится 🚀    │
        └────────────────────────────────────────┘
```

- **Popover API** (Baseline newly, 2025-01) — top-layer, `light-dismiss`, `Esc`, фокус-менеджмент бесплатно.
- ⚠️ **CSS Anchor Positioning — Baseline `limited`, Chromium-only. НЕ полагаться.** Позиционирование — обычным `position: absolute` относительно кнопки + фолбэк «прижать к правому краю панели». Anchor — только как прогрессивное улучшение через `@supports (anchor-name: --x)`.
- ⚠️ Данные эмодзи грузятся **лениво** при первом открытии (`await import('../emoji/data')`), с скелетоном. Не в основной бандл.
- Переключатель «символ / шорткод» — sticky, живёт в `settings.emojiInsertMode`, и он **знает про площадку**: при `target = jira | telegram` радио «Шорткод» дизейблится с пояснением (не молча — см. §5).

### 2.5 ToolDrawer → «Найти и заменить» (regex)

```
├────────────────────────────────────────────────────────────────────────────┤
│ ┏━━━━━━━━━━━━━━━━━┓                                                    ⌄    │
│ ┃🔎 Найти и заменить┃  ⇄ Транслитерация   📊 Статистика                     │
│ ┗━━━━━━━━━━━━━━━━━┛                                                         │
│                                                                            │
│ Найти     [ (\d{4})-(\d{2})-(\d{2})                    ]  ☑ Regex          │
│ Заменить  [ $3.$2.$1                                   ]  ☐ Aa  ☑ Слово ⓘ  │
│                                                                            │
│ Флаги: ☑ g  ☐ i  ☑ m  ☐ s  ☑ u  ☐ v(unicodeSets)              [шпаргалка ▾]│
│                                                                            │
│ ✓ 4 совпадения · группы: 1=год 2=месяц 3=день                              │
│                                                                            │
│ Предпросмотр замен (применится 4):                                         │
│ ┌────────────────────────────────────────────────────────────────────────┐ │
│ │ стр. 3   2026-07-14         →  14.07.2026                              │ │
│ │ стр. 9   2026-06-01         →  01.06.2026                              │ │
│ │ …ещё 2                                                                 │ │
│ └────────────────────────────────────────────────────────────────────────┘ │
│                                                                            │
│ [◀ Пред] [След ▶]        [Заменить]  [Заменить всё (4)]        [Отмена]    │
└────────────────────────────────────────────────────────────────────────────┘
```

- Совпадения подсвечиваются **прямо в `<textarea>`-оверлее** через **CSS Custom Highlight API** (`::highlight(cw-match)` / `::highlight(cw-current)`), а не тысячами `<span>`. Тот же приём, что в `devdata` (§2.5 PLAN.md (Часть II)). ⚠️ В `::highlight()` доступны только `color`, `background-color`, `text-decoration`, `text-shadow` — **`font-weight` менять нельзя**; текущее совпадение отличаем `background-color` + `text-decoration: underline`.
- ⚠️ Матчинг — **только в Web Worker** с таймаутом. Главный поток ничего не компилирует пользовательским `RegExp`. См. §5.3 и §8.1.
- «Заменить всё» пишет в черновик **одной undo-транзакцией** (`Ctrl+Z` откатывает 4 замены разом, а не по одной).

## 4. Потоки

### 4.1 Баг-репорт из шаблона → GitLab (основной)

1. Кликаю иконку → открывается **side panel** (S1) поверх страницы с багом. Черновик — последний активный.
2. `📋 Шаблоны → Баг-репорт → Вставить: Заменить черновик` (пустой — подтверждения не спрашиваем; непустой — спрашиваем, см. §5.2).
3. Пишу шаги. `Ctrl+Shift+C` → `- [ ]`. `Ctrl+Shift+D` → `<details>` для логов.
4. **«⧉ Вставить окружение»** → превью блока → вставка таблицы. URL берётся из активной вкладки (`activeTab`, по жесту).
5. Скриншот? — **🔴 нет.** Это `capture`/`export`, не мы. В шаблоне на этом месте плейсхолдер `<!-- перетащите скриншот в поле GitLab после вставки -->` и подсказка: «Картинки вставляются на стороне GitLab — Markdown их только ссылается».
6. Площадка уже `GitLab` → CounterStrip показывает `GitLab ✓ совместимо`.
7. **[Копировать для GitLab]** → `ClipboardItem` c `text/plain` (GLFM). Тост: «Скопировано для GitLab — 402 Б. Чекбоксы и `<details>` сохранены».
8. `Ctrl+V` в поле GitLab.

### 4.2 Выделил текст на странице → «Добавить в черновик»

1. Выделяю сообщение об ошибке на странице → ПКМ → **«Добавить выделенное в черновик»**.
2. Background: `activeTab` уже выдан жестом контекстного меню → `scripting.executeScript` читает `getSelection().toString()` → дописывает в конец активного черновика (или как `>`-цитату, по настройке) + строку-источник `— [Заголовок](URL)`.
3. Панель открывается (жест есть) и **скроллит к вставленному фрагменту**, подсвечивая его на 1.5 с (`::highlight()`, не анимация фона).
4. Если панель уже открыта — обновляется живьём. Если черновика нет — создаётся «Из выделения — <хост>».
5. ⚠️ Ничего на странице не меняется. `activeTab` истекает — это нормально, второй вызов = второй жест.

### 4.3 Find & Replace с regex

1. `Ctrl+F` (внутри редактора) → ToolDrawer, вкладка «Найти и заменить», фокус в поле «Найти».
2. Ввожу `(\d{4})-(\d{2})-(\d{2})`, `☑ Regex`. На каждый keystroke (debounce 150 мс) → **сообщение в Worker**: `{pattern, flags, text}`.
3. Worker компилирует и матчит. Главный поток ставит таймер `regexTimeoutMs`.
4. Ок → `{matches: [[start,end,groups]…]}` → подсветка через Custom Highlight API + счётчик «4 совпадения» + именованные группы в подсказке.
5. Ввожу `$3.$2.$1` → **предпросмотр замен до применения** (список «было → станет», первые 5 + «ещё N»).
6. «Заменить всё» → снимок ⚑ → одна undo-транзакция → тост «Заменено 4».
7. Ошибка/таймаут → §5.3/§5.4.

### 4.4 Транслитерация фамилии в slug для имени ветки

1. Выделяю `Щербаков Юлия` в черновике → ToolDrawer → «Транслитерация» (`Ctrl+Shift+L`).
2. Источник авто-становится «Выделение (14 симв.)».
3. Вижу **все пять стандартов сразу на своём тексте** → выбираю `slug`.
4. Результат `shcherbakov-yuliya`, лимит 63 не превышен.
5. **[Заменить выделение]** — текст в черновике меняется; либо **[Копировать]** — если это имя ветки, а не часть текста.

### 4.5 Смена площадки и копирование

1. `Куда: [GitLab ▾] → Slack`.
2. Превью **немедленно перерисовывается в mrkdwn-режиме** (это единственное, что меняется в UI мгновенно — сам черновик не трогается).
3. CounterStrip: `⚠️ Slack: 2 конструкции упростятся` → клик → §6.4 (диалог совместимости).
4. **[Копировать для Slack]** → конверсия на выходе: `**bold**` → `*bold*`, `- [ ]` → `• ☐`, `<details>` → плоский текст с заголовком, таблица → моноширинный блок.
5. Тост: «Скопировано для Slack. Упрощено: таблица → блок кода, `<details>` → развёрнут.»

🔴 **Инвариант:** исходный Markdown в `local:drafts` **никогда не переписывается конвертером**. Конверсия происходит только в момент копирования. Переключение площадки туда-обратно **лосслесс**.

### 5.2 Несохранённое / автосейв

- Индикатор в шапке черновика: `● Сохраняется…` → `✓ Сохранено 14:03`. **Никогда** не показываем «Сохранено», пока `storage.local.set` не зарезолвился (⚠️ ровно этой ложью выстрелил `blur` c `sync`: UI показывал «saved», данные не долетали).
- Ошибка записи → `⚠️ Не удалось сохранить (QuotaExceeded). Текст в редакторе цел. [Освободить место] [Экспортировать .md]` — и **не даём тихо потерять** (§8.2).
- Деструктивные операции при непустом черновике (вставка шаблона с заменой, восстановление снимка, импорт) → `<dialog>`-подтверждение с указанием, сколько символов будет перезаписано.

### 5.3 Regex: таймаут

```

│ вложенные повторы вроде (a+)+ — они перебирают комбинации экспоненциально.  │
│ Поиск выполняется в отдельном потоке и был прерван, поэтому редактор        │

│  • убрать вложенный квантификатор:  (a+)+  →  a+                           │

После таймаута воркер **терминируется и пересоздаётся** — следующий запрос идёт в свежий. Кнопка «Повторить» доступна сразу.

│        ⚠️ Неверный шаблон: не закрыта скобка «(» (позиция 1)               │

### 5.5 Огромный черновик

- Порог **200 000 графем** (≈ 400 КБ): баннер `ⓘ Большой черновик. Живое превью переведено в ручной режим — [Обновить превью]` + автоотключение подсветки regex по всему тексту (только по видимому окну).
- ⚠️ Превью **не рендерим на каждый keystroke** и на маленьких: debounce 120 мс + `requestIdleCallback`. Парсинг+санитайз > 50 мс → авто-переход в ручной режим с честным сообщением, а не молчаливый лаг.
- `content-visibility: auto` на блоках превью.
- Порог **2 МБ**: `⚠️ Черновик близок к пределу хранилища. [Экспортировать и очистить]`.

Источник истины — **внутренний GFM-супермножество** (GFM + `<details>`). Копирование = `convert(draft, target)`.

| Целевая | Действия конвертера |

### 6.3 Правило «не врать»

Конвертер **никогда не выкидывает содержимое молча**. Всё, что нельзя выразить в целевом синтаксисе, — **деградирует с сохранением текста** (таблица → код-блок, details → развёрнутый текст), а факт деградации показывается **до** копирования.

## 7. Безопасность

### 7.1 🔴 Где проходит граница

Превью — это HTML, вставленный **в страницу расширения**, у которой есть `chrome.*` и доступ к `storage` с чужими черновиками. Текст приходит из буфера, со страницы, из чужого баг-репорта. **XSS здесь = компрометация расширения, а не дефейс.**

Пайплайн (единственный разрешённый):

```
draft(text)
  → markdown-it({ html: true, linkify: false, typographer: false })   // html:true ОБЯЗАТЕЛЕН — нужен <details>
  → HTML-строка (СЧИТАЕТСЯ ВРАЖДЕБНОЙ)
  → DOMPurify.sanitize(html, { RETURN_DOM_FRAGMENT: true, ALLOWED_TAGS, ALLOWED_ATTR, ... })
  → DocumentFragment
  → previewEl.replaceChildren(fragment)        // ✅ узлы, НЕ строка
```

🔴 **`innerHTML` / `outerHTML` / `insertAdjacentHTML` / `dangerouslySetInnerHTML` в кодовой базе запрещены**, включая «безопасные» места. Правило механическое, не «по внимательности»: ESLint-запрет + code review. Причина: единственный способ гарантировать, что санитайзер нельзя обойти, — не иметь ни одной точки, где строка становится DOM мимо него. `RETURN_DOM_FRAGMENT` + `replaceChildren` эту точку убирает физически.

⚠️ `linkify: false` — не косметика: автолинковка превращает произвольный текст пользователя в `href`, расширяя поверхность. Ссылки — только явные `[t](u)`.

### 7.2 Allow-list (явный, не «запретить плохое»)

| | Значение |
|---|---|
| `ALLOWED_TAGS` | `p br hr h1 h2 h3 h4 h5 h6 strong em del s code pre blockquote ul ol li a img table thead tbody tr th td kbd sup sub details summary span div input` |
| `ALLOWED_ATTR` | `href src alt title align colspan rowspan class type checked disabled open lang dir` |
| `ALLOWED_URI_REGEXP` | **только** `^(?:https?|mailto):` — 🔴 `javascript:`, `data:`, `blob:`, `vbscript:` отрезаны |
| Запрещено безусловно | `script style iframe object embed form link meta base svg math template`, **все `on*`**, `srcdoc`, `formaction`, `xlink:href`, `style` как атрибут |
| `FORBID_CONTENTS` | `script style` |
| `input` | только `type="checkbox"` + `disabled` (это чекбоксы GFM). ⚠️ Хук `afterSanitizeAttributes` принудительно ставит `disabled` — иначе получаем интерактивный ввод в привилегированной странице |
| `a` | хук проставляет `target="_blank" rel="noopener noreferrer"`. `img` — `loading="lazy"` |

⚠️ **Ошибка «просто разрешить `class`»**: `class` в allow-list мы держим, но превью рендерится **в Shadow DOM** (`closed`) со своими стилями — чужой `class` не может дотянуться до стилей расширения и подделать UI (clickjacking «Копировать»).
⚠️ **`style`-атрибут — 🔴 нет.** Он позволяет нарисовать поверх кнопок расширения фейковый UI. Цена — превью чуть беднее. Правильный размен.

⚠️ **Открытый вопрос (§11 PLAN.md (Часть II)): нативный `Element.setHTML()` / Sanitizer API.** Если к моменту реализации он Baseline — он предпочтительнее внешней зависимости. **Дефолт остаётся DOMPurify** (dual MPL-2.0 / Apache-2.0), а пайплайн спроектирован так, чтобы санитайзер был **одной подменяемой функцией** `sanitizeToFragment(html): DocumentFragment`.

### 7.3 Что происходит с вырезанным — и что видит пользователь

DOMPurify-хук `uponSanitizeElement`/`uponSanitizeAttribute` копит `removed[]`. Если непусто — **баннер над превью** (не тост: тост исчезнет, а пользователь должен понимать, что превью неполно):

```
┌────────────────────────────────────────────────────────────────────────────┐
│ ⚠️ В превью не показаны 2 фрагмента                                        │
│                                                                            │
│ Мы вырезали HTML, который может выполнить код внутри расширения:            │
│   • <script> (1)                                                           │
│   • атрибут onerror у <img> (1)                                            │
│                                                                            │
│ Ваш текст НЕ изменён — вырезано только из превью. При копировании           │
│ отправится то, что вы написали.                                            │
│ GitHub и GitLab вырежут это тоже: их собственные санитайзеры строже нашего. │
│                                            [Показать, что вырезано ▾]       │
└────────────────────────────────────────────────────────────────────────────┘
```

🔴 Ключевые свойства этой копии:
1. **Черновик не трогаем.** Мы не «чиним» текст пользователя — это его текст. Санитизируется только рендер.
2. **Не обещаем, что копия безопасна для площадки** — говорим, что *площадка тоже вырежет*. Это правда и полезно.
3. «Показать, что вырезано» рендерит удалённое **как экранированный текст в `<pre>`** (`textContent`, никогда не HTML).

### 7.4 Почему нет сети

- 🔴 Нет `host_permissions`, нет `optional_host_permissions`, нет `fetch`. CSP страниц расширения: `script-src 'self'; object-src 'self'; connect-src 'none'` — `connect-src 'none'` делает сеть **невозможной технически**, а не «по договорённости».
- Эмодзи, словари транслита, парсер — **всё в бандле**. Никакого CDN (🔴 remote code, §10.7 PLAN.md (Часть II) — и WASM тоже считается кодом).
- Копия в UI (About + первый запуск): **«Ваш текст остаётся в браузере. Расширение не имеет доступа к сети: ни одного сетевого запроса, ни аналитики, ни облачной синхронизации, ни ИИ-сервисов.»**
- AMO: `data_collection_permissions: { required: ['none'] }`.
- ⚠️ С 1 августа 2026 CWS требует, чтобы **любой** сбор был prominently disclosed **в UI**. У нас сбора нет — и это надо не просто «не делать», а **написать**, потому что категория «текстовые редакторы» полна расширений, которые сливают набранное.

## 8. Отказоустойчивость

### 8.1 Worker и ReDoS

```
main                          worker (regex.worker.ts)
 │ postMessage({id, pattern, flags, text})
 │ setTimeout(TIMEOUT) ────────────────────┐
 │                              compile+match (может зависнуть навсегда)
 │ ◀── {id, matches} ───────────  ok       │
 │ clearTimeout                            │
 │                                         ▼ таймер сработал
 │ worker.terminate()          ✂ поток убит принудительно
 │ worker = spawn()            новый воркер готов к следующему запросу
 │ UI → §5.3
```
- 🔴 `new RegExp(userInput)` **никогда не выполняется в главном потоке** — даже для «проверить, валиден ли». Валидацию синтаксиса тоже делает воркер (`SyntaxError` возвращается сообщением): иначе патологический паттерн повесит UI ещё на этапе компиляции.
- Один in-flight запрос; новый keystroke отменяет предыдущий (`id`-гейт, старые ответы игнорируются).
- Воркер **не имеет доступа к `storage`** — он чистая функция «текст+паттерн → диапазоны». Компрометация воркера ничего не даёт.
- Транслит и подсчёт графем — **синхронно в главном потоке** (линейные, без бэктрекинга). ⚠️ Кроме «Транслитерировать весь черновик» > 100 КБ — туда же в воркер, чтобы не ловить long task.

### 8.2 Квота хранилища

- `storage.local` — ~10 МБ (`unlimitedStorage` **не берём**: лишнее разрешение ради того, чего не бывает).
- Перед каждой записью — оценка размера. При > 80 % → баннер `Занято 8.1 из 10 МБ. Старые снимки будут вытесняться. [История] [Экспорт]`.
- `QuotaExceededError` → 🔴 **не терять текст**: (1) текст остаётся в `<textarea>` и в `session:unsaved`; (2) баннер с кнопками «Удалить старые снимки» / «Экспортировать .md»; (3) статус в шапке — **«Не сохранено»**, никогда не «Сохранено».
- Кольцевой буфер снимков вытесняет автосейвы, но **не** ⚑-снимки перед деструктивными операциями.

### 8.3 Автосейв и восстановление после падения

- `input` → debounce `autosaveDelay` (800 мс) → запись в `local:drafts` (RMW под `navigator.locks`, как в `blur/utils/storage.ts` — иначе панель и background затрут друг друга).
- Дополнительно — **синхронный** сброс в `session:unsaved` на `visibilitychange` / `pagehide` панели (закрытие сайдбара = уничтожение документа).
- На старте: если `session:unsaved` новее `local:drafts[active].updatedAt` → баннер `Восстановлен несохранённый текст (14:03). [Оставить] [Отменить восстановление]` — с возможностью откатить, а не «мы уже всё за вас решили».

### 8.4 Смерть service worker'а

SW у нас почти ничем не занят (контекстное меню + открытие панели). Правила:
- 🔴 **Никакого состояния в памяти SW.** Черновик читается/пишется UI-документом напрямую.
- `contextMenus.create` — только в `runtime.onInstalled` (пересоздание на каждый запуск SW плодит дубликаты/ошибки `duplicate id`).
- Если панель закрыта в момент клика по контекстному меню — background дописывает текст в `local:drafts` **сам** (под тем же локом) и открывает панель. Порядок: сначала запись, потом открытие — иначе панель отрендерит старый текст.

### 8.5 Прочее

| Отказ | Поведение |
|---|---|
| Парсер markdown бросил исключение | Превью → `⚠️ Не удалось отрендерить превью. Текст цел. [Подробности]`. Редактор жив |
| Эмодзи-данные не подгрузились (chunk load error) | Пикер: `Не удалось загрузить набор эмодзи. [Повторить]`; можно вставить символ вручную. Редактор жив |
| Custom Highlight API отсутствует | Фолбэк: список совпадений в панели + скролл-к-совпадению без подсветки. **Не** городить `<span>`-подсветку |
| `Intl.Segmenter` отсутствует (не должно; Baseline) | Считаем по code points и **честно пишем** «приблизительно», а не молча даём `.length` |

| Превью вкл/выкл (узкий: переключить таб) | `Ctrl+Shift+P` | |
| Копировать для площадки | `Ctrl+Enter` | Главное действие |
| Копировать как HTML | `Ctrl+Shift+Enter` | |
| Отмена / возврат | `Ctrl+Z` / `Ctrl+Shift+Z` | Своя undo-стопка (нативная у `<textarea>` ломается при программной вставке) |
| Закрыть drawer/popover/dialog | `Esc` | |
| Отступ списка | `Tab` / `Shift+Tab` **только когда каретка в списке** | ⚠️ Иначе `Tab` **обязан** уводить фокус — перехват `Tab` в `<textarea>` = ловушка фокуса для клавиатурных и скринридерных пользователей. Ловушку не делаем никогда |

- ⚠️ Ни один шорткат не объявлен в `manifest.commands` кроме одного глобального «открыть черновик» — остальные живут **внутри документа** (§1.1: у инструментов нет своих точек входа, а глобальный шорткат = точка входа).
- Все шорткаты перечислены в `[?]`-панели (`<dialog>`), открывается по `F1` / `?`.

### 9.2 Семантика

- ToolDrawer — `role="tablist"` / `role="tab"` / `role="tabpanel"`, `aria-selected`, `aria-controls` **только на активном табе** (конвенция дома, `PLAN.md` §18c), стрелки ←/→ по табам.
- Результаты поиска, автосейв, «скопировано», предупреждения совместимости — `role="status" aria-live="polite"`. Ошибки (regex invalid, quota, clipboard fail) — `role="alert"` / `aria-live="assertive"`.
- ⚠️ Счётчик **не** `aria-live` — иначе скринридер зачитывает числа на каждый символ. Он `aria-label`-описан и озвучивается по запросу/фокусу.
- Превью — `role="region" aria-label="Превью — близко к GitHub, не идентично"`.
- Все иконочные кнопки — `aria-label` + видимый `title`; ⚠️ подсказки — **видимым текстом**, а не только `title` (`PLAN.md` §18c, SEO-урок).
- `:focus-visible` с 2px `outline: var(--accent)` на всех интерактивных элементах, включая `summary` в превью.
- Severity — **не только цветом**: `⚠️`/`✗`/`✓` + слово.
- `prefers-reduced-motion` → View Transitions и анимации drawer'а выключены.

### 9.3 Тема

- Токены — **байт-в-байт из `packages/ui`** (`--bg --fg --sub --line --card --accent --accent-fg --warn --warn-bg --ok --radius`). ⚠️ `packages/ui` — долг из `PLAN.md` §18, и по §7 PLAN.md (Часть II) он делается **до** этой волны. `compose` его потребитель, а не пятый форк токенов.
- `color-scheme: light dark` на `:root` — иначе нативные `<select>`, скроллбары и **`<textarea>`** в тёмной теме останутся светлыми (ровно этот баг ловили в §18a).
- Переключатель Auto/Light/Dark → `:root[data-theme='light'|'dark']` (specificity 0,2,0 бьёт `prefers-color-scheme`-фолбэк). Синхронный сид темы из `localStorage` **до** `createRoot` — иначе FOUC (урок `seo`, §18c).
- Превью в Shadow DOM наследует токены через `:host` + `inherit`; GitHub-подобные стили превью — свои, но на наших токенах (никаких хардкод `#fff`).
- Контраст: `--sub` держит 4.5:1 на `--bg`, `--card` **и** `--warn-bg` (уже выверено в `blur`).

### 10.2 Библиотеки

| Задача | Выбор | Лицензия | Обоснование |
|---|---|---|---|
| **Markdown → HTML** | **`markdown-it`** | **MIT** | Плагинная архитектура (нужна: GFM-таски, `<details>`-friendly `html: true`), стабильный AST-подобный token-stream — а он нужен **и для конвертеров площадок**, чтобы не писать пять regex-трансформеров по сырому тексту. `marked` (MIT) быстрее и легче, но его расширяемость слабее, а нам нужен один парсер на превью + 6 конверторов. ⚠️ Оба **не санитайзят** |

| **Эмодзи-данные** | **`emojibase-data`** (или `unicode-emoji-json`) | **MIT** / CC-BY-4.0 (данные) | ⚠️ **Проблема веса — главная.** Полный набор с шорткодами и переводами — **сотни КБ**. Решение: (1) отдельный chunk через `await import()` — **никогда** в основной бандл; (2) грузим **только** `en` + shortcodes, без skin-tone-вариаций и без описаний; (3) индекс поиска строим в рантайме, а не бандлим. Бюджет: **основной бандл < 250 КБ gzip**, эмодзи-chunk отдельно |
| **Транслитерация** | ⚠️ **Свои таблицы, а не библиотека** | — | Готовые npm-пакеты обычно реализуют **один** «средний» стандарт и врут на краях (`ь`, `ъ`, `й` в конце, `е` после гласной). Нам нужно **пять** стандартов, различия между которыми — и есть продукт (§2.6). Это ~5 таблиц × ~40 правил + контекстные правила = меньше 5 КБ своего кода, полностью тестируемого. Тянуть зависимость, которую всё равно придётся исправлять, — хуже |
| Slug | Своё (после транслита) | — | `slugify`/`speakingurl` тянут свои таблицы и дублируют пункт выше |
| Highlight кода в превью | ⚠️ **Не берём Prism/highlight.js** | — | +200 КБ ради превью. В v1 блок кода — моноширинный без раскраски; при необходимости — Custom Highlight API + свой мини-токенайзер (как в `devdata`) |
| Mermaid | 🔜 v2, **лениво** | MIT | Тяжёлая. Только `await import()` и только по явному «Отрендерить диаграмму» |

**Бандл-бюджет:** основной чанк ≤ 250 КБ gzip (React + markdown-it + DOMPurify + свой код). Эмодзи, mermaid — отдельные ленивые чанки. ⚠️ Регресс бюджета = блокер релиза: на 320 px панели медленный старт заметен сразу.

## 11. Что НЕ делать

| Анти-фича | Почему 🔴 / ⚠️ |
|---|---|
| 🔴 **AI «улучшить текст» / «переписать»** | Сеть + отправка пользовательского текста наружу + сбор данных + **другая цель**. Убивает zero-permission позиционирование и single purpose одним движением. Ни локального «маленького» LLM (вес, WASM-ревью на AMO), ни API-ключей |
| 🔴 **Облачная синхронизация черновиков** | То же самое. Кросс-девайс — через «Экспорт .md» / «Импорт» |
| 🔴 **Отдельная иконка/попап/шорткат/пункт меню у regex, транслита или счётчика** | ⚠️ **Единственное, на чём держится single purpose.** Получил свою точку входа — стал отдельным продуктом → bundle → отказ |
| 🔴 **Транслитерация/regex по тексту СТРАНИЦЫ** (не черновика) | Это уже модификация/обработка чужих страниц. Другая цель, другие разрешения, другой продукт |
| 🔴 **Конвертация страницы в Markdown** | Это `export` (§3 PLAN.md (Часть II)). Мы принимаем только вставленный/выделенный текст |
| 🔴 **Скриншоты, вставка изображений, загрузка картинок** | Это `capture`. Markdown только **ссылается** на картинки; загружает их площадка |
| 🔴 **`innerHTML` где угодно, включая «доверенные» места** | §7.1. Одна точка обхода санитайзера = компрометация расширения |
| 🔴 **Аналитика/телеметрия/Sentry «чтобы видеть ошибки»** | Черновик = пользовательский текст, часто с внутренними URL, токенами из логов, именами клиентов. С 1 авг. 2026 любой сбор = prominent disclosure в UI и «strictly necessary». У нас — ноль |
| 🔴 **`unlimitedStorage`, `downloads`, `tabs`, `<all_urls>`** | Ни одно не нужно: экспорт — через `<a download>` из документа расширения (`export` §3.1), URL страницы — через `activeTab` по жесту. Каждое лишнее разрешение = install-предупреждение, которого у нас сейчас **ноль** |
| 🔴 **Автооткрытие панели / автовставка буфера при старте** | Требует чтения буфера без жеста (нельзя) и раздражает. Плюс `clipboardRead` — это уже предупреждение |
| ⚠️ **WYSIWYG-режим** («печатаю жирным, вижу жирное») | Соблазнительно и убивает продукт: WYSIWYG над Markdown требует полноценного contenteditable-редактора (ProseMirror/Lexical, +150–300 КБ) и порождает нескончаемые баги нормализации. Наш пользователь — инженер, который **хочет видеть исходник**, потому что вставляет исходник |
| ⚠️ **Свой словарь орфографии** | Нативный `spellcheck` — бесплатно. Свои словари — мегабайты. Внешний сервис — 🔴 |
| ⚠️ **Обещание «превью как на GitHub»** | Мы не воспроизведём их санитайзер и расширения байт-в-байт. В UI написано «близко, не идентично» (§2.2, §9.2) — и это надо **сохранить при копирайтинге листинга**, а не потерять в маркетинге |
| ⚠️ **Кнопка «Скачать .md» через `downloads`** | Blob + `<a download>` делает то же самое без разрешения. Брать `downloads` только если реально понадобится `saveAs`-диалог |
| ⚠️ **Панель `sidePanel.setOptions({tabId})` (черновик на вкладку)** | Нет аналога в Firefox + продуктово неверно: черновик не должен исчезать при смене вкладки |

## 12. Открытые вопросы (проверить перед фазой реализации)

- **WXT: `side_panel` (Chrome) vs `sidebar_action` (Firefox) из одного entrypoint** — проверить на живой сборке **до** написания UI (§11 PLAN.md (Часть II)). Если WXT не разводит их автоматически — два тонких HTML-шелла над одним React-приложением.
- **`Element.setHTML()` / Sanitizer API — Baseline?** Если да — заменяет DOMPurify внутри `sanitizeToFragment()`. Иначе DOMPurify.
- **GLFM vs GFM в `<details>` и чекбоксах** — проверить вживую; от этого зависит, показываем ли предупреждение при `target = gitlab`.
- **Custom Highlight API на 400 КБ текста** при частом пересоздании `Range` — профилировать (тот же вопрос стоит у `devdata`).
- **Вес эмодзи-набора** после tree-shaking (`emojibase-data` только `en` + shortcodes) — замерить до принятия в бандл.
- Лицензия данных эмодзи (CC-BY-4.0 у emojibase) → строка в `THIRD-PARTY-NOTICES.md`.
- **Клипборд в Firefox sidebar**: `navigator.clipboard.write()` с `text/html` — проверить, что не требует `dom.events.asyncClipboard.clipboardItem`-пref и работает без `clipboardWrite`-фолбэка.
