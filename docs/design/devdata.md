## 0. Итог одним экраном

| Решение | Значение |
|---|---|
| **Главная поверхность** | **Полностраничная вкладка инструмента** `tool.html` (extension page). Клик по иконке открывает popup-лаунчер 320px, но вся работа — в полной вкладке |
| Поверхности всего | popup-лаунчер · **tool page** (табы: Данные · JWT · Схема · Настройки) · `options_ui` → **та же** tool page на `#/settings` · контекстное меню |
| DevTools-панель | 🔴 **нет в v1.** Обоснование в §1.4 |
| Side panel | 🔜 v2. ⚠️ `side_panel` (Chrome) vs `sidebar_action` (Firefox) — открытый вопрос PLAN.md (Часть II) §11, не блокировать им v1 |
| Разрешения (install-time) | `storage`, `contextMenus`, `activeTab` — **ноль install-предупреждений** |
| Разрешения (optional, по кнопке) | `scripting` (без host!) — «форматировать эту вкладку по клику»; `<all_urls>` — только для **авто**-формата при навигации |
| Сеть | **ноль**. Ни fetch, ни JWKS-URL, ни аналитики. CSP без `connect-src` |
| Персист | `sync:prefs` (≤2 КБ) · `local:document` (≤1 МБ, опц.) · **JWT и secret — только RAM** |

## 1. Информационная архитектура

### 1.1 Правило, из которого выводится всё остальное

PLAN.md (Часть II) §2.1: *«Всё в одном UI (табы одного инструмента), а не три продукта в общей оболочке»*. Отсюда жёсткий инвариант, который надо защищать при каждом ревью PR:

> 🔴 **Ни одна функция не имеет собственной точки входа.** Нет иконки «JWT decoder», нет отдельной страницы «YAML converter», нет второго пункта в меню расширений. Есть **один** инструмент с табами. Как только JWT получает свой попап — расширение становится bundle'ом (ср. `compose` §6.1: та же логика, тот же риск).

### 1.2 Поверхности и зачем каждая

| Поверхность | Файл | Зачем именно она | Почему нельзя без неё |
|---|---|---|---|
| **Tool page** (главная) | `entrypoints/tool/` → `tool.html` | Рабочая область: редактор + дерево + инспектор. Открывается во **вкладке**, живёт сколько нужно, переживает потерю фокуса, имеет полную ширину экрана и права extension-страницы (Worker'ы, `storage`) | Единственная поверхность, где помещается 50 МБ JSON и трёхколоночный layout |
| **Popup-лаунчер** | `entrypoints/popup/` | 320px, ~5 секунд жизни. Три действия: **Вставить и открыть**, **Открыть инструмент**, **Форматировать JSON на этой вкладке**. Плюс тумблер темы | Клик по иконке — единственный жест, который выдаёт `activeTab`. Без popup нет контекста «текущая вкладка» вообще |
| **Options** | `options_ui.page = "tool.html#/settings"` | Пункт «Параметры» в меню браузера должен куда-то вести. Он ведёт **в тот же инструмент**, на таб «Настройки» | Отдельная options-страница = вторая точка входа = дрейф к bundle'у. ⚠️ `open_in_tab: true`, иначе Firefox покажет настройки в узкой рамке about:addons |
| **Контекстное меню** | `background.ts` | «Открыть выделенное в Data Toolkit» (`contexts: ['selection']`) → `info.selectionText` **без инъекции скрипта** | Самый частый реальный сценарий: увидел JSON в логе/на GitHub → выделил → открыл |

**Почему главная — полная вкладка, а не popup и не side panel:**

1. **Popup закрывается при потере фокуса.** Любой клик мимо — и разобранные 8 МБ JSON, дерево, позиция скролла и невосстановимый in-memory secret исчезли. Для инструмента, в котором сидят минутами, это дисквалификация. (Тот же вывод у `capture` §1.1: «popup не годится — он закрывается и убивает поток».)
2. **Popup нельзя открыть без клика по иконке** — а сюда приходят из контекстного меню и по drag-drop файла.
3. **Ширина.** Дерево + сырой текст + инспектор значения — это три колонки. В 320px это не layout, а издевательство.
4. **Side panel** (v2) хорош ровно для одного сценария — «смотрю на страницу и параллельно ковыряю её JSON». Но: ⚠️ в Chrome это `side_panel`, в Firefox — `sidebar_action`, и как WXT разводит их из одного entrypoint, **не проверено** (PLAN.md (Часть II) §11). Ставить v1 на непроверенный ключ манифеста нельзя. Side panel добавляется как **третья проекция того же React-приложения** (тот же роутер, тот же store), а не как новый продукт.
5. **Мобильный.** PLAN.md (Часть II) §8: `devdata` — лучший мобильный кандидат. На Firefox Android popup — это и так полноэкранная страница, а side panel не существует. Полная вкладка деградирует туда идеально.

### 1.3 Навигация внутри tool page

```
Данные        ← ядро: редактор + дерево + конвертация. Формат автодетектится
JWT           ← отдельный таб, потому что это КРЕДЕНШЛ и у него другая рамка (§7)
Схема         ← JSON Schema: валидация текущего документа
Diff          ← 🔜 v2, в v1 таба нет вообще (не «disabled», а нет)
Настройки     ← сюда же ведёт options_ui
```

Формат (JSON / JSON5 / JSONC / YAML / XML / CSV) — **не таб**, а свойство документа: чип в шапке рабочей области с автодетектом и ручным override. Иначе шесть табов ×4 = двадцать четыре состояния, и пользователь теряется. Конвертация — **действие над документом** («Конвертировать в →»), а не отдельное место.

### 1.4 Чего нет и почему

- 🔴 **DevTools-панель.** Соблазн: «показывать JSON-ответы из Network». Но вопрос «что происходит в сети на этой странице» — это `perf` и `assets` (PLAN.md (Часть II) §4.3 проводит ровно эту границу). `devdata` отвечает на вопрос «вот данные, которые **у меня в руках** — покажи и преобразуй». Панель размывает цель и создаёт вторую точку входа. Кроме того `chrome.devtools.network` даёт тела ответов — это уже перехват трафика, и для «инспектора данных» ревьюер справедливо спросит зачем. 🔜 v3 вместе с `.har`-вьюером, и тогда пересматривать эту границу целиком.
- 🔴 **Omnibox-ключевое слово.** Вторая точка входа, нулевая польза.
- 🔴 **Badge на иконке** («на этой странице JSON»). Требует знать `document.contentType` на каждой странице → `<all_urls>` у всех, а не у согласившихся. Ровно та ловушка из §2.2 PLAN.md (Часть II).

```
│   "users": [                 │   - id: 12345678901234567890                    │
│     { "id": 123…,            │     name: Иван                                  │
│       "name": "Иван" }       │     active: true                                │
│   ]                          │                                                 │
├──────────────────────────────┴─────────────────────────────────────────────────┤
│ ⚠ 3 предупреждения преобразования                                    [Развернуть]│
│   • CSV пропущен: документ не является плоским массивом объектов               │
│   • XML: ключ «2fa» не может быть именем тега (начинается с цифры) → «_2fa»    │

```

⚠️ **Панель предупреждений преобразования обязательна.** JSON→XML и JSON→CSV — операции **с потерями**, и молча терять данные — это ровно «врать в UI». Список известных потерь: `null` в XML, типы в CSV, вложенность в CSV, дубли ключей, порядок ключей, `-0`, `NaN`/`Infinity` в YAML, комментарии JSONC (теряются при любой конвертации в JSON).

## 4. Потоки

### 4.1 Вставить JSON → посмотреть

1. Popup → «Вставить из буфера и открыть», либо tool page открыта и `⌘V` в неё.
2. ⚠️ `navigator.clipboard.readText()` в **popup** может отказать (нет фокуса/нет transient activation) — из клика по кнопке отработает, но на всякий случай **фолбэк**: открываем tool page **с пустым редактором и фокусом в нём**, показываем подсказку «Нажмите ⌘V». Никогда не показываем «Ошибка буфера обмена» как тупик.
3. Текст → **Worker**: детект формата (BOM → `{`/`[` → `---`/`:` → `<` → разделители → 3 сегмента base64url) → парс.
4. < 256 КБ: парс синхронно (Worker всё равно, но результат приходит за один тик) → рендер.
5. Результат: дерево + текст + шапка со статистикой.

**Ошибки:**
- Пусто → остаёмся на empty state, не показываем ошибку.
- Не парсится ни одним парсером → §5.4 (экран «не распознали»).
- Похоже на JWT (3 сегмента base64url, header с `alg`) → **не парсим как данные**, а предлагаем: «Похоже на JWT. Открыть в табе JWT?» — с кнопкой, а не автоматически (автопереход на экран с большим красным предупреждением про креденшлы дезориентирует).

### 4.2 Drag & drop файла

1. Drop-зона — **всё окно tool page** (`dragover` → оверлей на весь экран, не только на пустую область: попасть в маленький прямоугольник мышью с файлом — мука).
2. `dataTransfer.files[0]` → проверка размера:
   - ≤ 50 МБ → `file.stream()` в Worker, парс.
   - 50–200 МБ → ⚠️ **предупреждение с явным согласием**: «Файл 120 МБ. Инструмент рассчитан на 50 МБ. Может подвиснуть на десятки секунд или упасть по памяти. Открыть всё равно? [Открыть] [Отмена]». Не отказ — предупреждение. Разработчик знает, что делает.
   - > 200 МБ → отказ с числом: «Файл 640 МБ. Это больше, чем расширение может разобрать в браузере». Не «слишком большой файл», а конкретика.
3. Расширение файла → **подсказка** для детектора, не приговор (`.txt` с JSON внутри разбирается).
4. **Несколько файлов** → v1: берём первый, говорим об этом («Открыт `a.json`. Инструмент работает с одним документом за раз»). Молча брать первый из десяти — тоже враньё.

### 4.3 Открыть .json-страницу → отформатировать

**Путь A (без host-доступа, дефолт):**
1. Клик по иконке → popup. `activeTab` выдан.
2. «Форматировать JSON на этой вкладке».
3. Если `scripting` ещё не выдан → `permissions.request({permissions:['scripting']})`. ⚠️ Это **permission-only** запрос: браузер не показывает страшное «читать данные на всех сайтах», потому что host-доступ не запрашивается — его на эту вкладку и на этот момент уже дал `activeTab`. ⚠️ **Firefox MV2**: `scripting` не нужен вовсе — там `tabs.executeScript` работает от `activeTab`. Кнопка на Firefox активна сразу, без промежуточного запроса.
4. `scripting.executeScript` → скрипт читает `document.contentType` и `document.body.innerText`.
5. `contentType` = `application/json` / `+json` / `text/plain` с валидным JSON → рендерим вьюер поверх (§2.12).
6. Не JSON → ⚠️ **не молчим**: тултип в popup «На этой странице нет JSON-документа (тип: `text/html`). Скопируйте нужный фрагмент и вставьте в инструмент».

**Путь B (авто, opt-in):** §2.11 → грант → регистрируем content-script на `document_start` → при навигации на `application/json` подменяем вид.

**Ошибки:**
- `chrome://`, `about:`, страница Web Store, `view-source:` → инъекция физически невозможна. Кнопка **disabled** с текстом «Браузер не разрешает расширениям работать на этой странице», а не «Ошибка».
- ⚠️ Пользователь отклонил `permissions.request` → кнопка остаётся disabled, поясняющий текст, кнопка «Попробовать снова». Никаких повторных автоматических запросов.
- Firefox + встроенный viewer: наш скрипт **отработает**, но пользователь увидит виджет Firefox. Показываем однократную (dismissible) плашку с инструкцией `about:config`. ⚠️ Не пытаться «побороть» — это гонка, которую мы проиграем, и обход настройки браузера.

### 4.4 Декодировать JWT

1. Таб JWT (или popup → вставка, детект 3-сегментного base64url → предложение).
2. Вставка → **декод мгновенно, синхронно, без кнопки**. Декод — это `atob` + `JSON.parse`, ~20 строк (PLAN.md (Часть II) §2.3), Worker не нужен.
3. Header, payload, расшифровка claims, статус `exp`/`nbf`/`iat`.
4. Верификация — **только по явной кнопке** (это осознанное действие с ключом, а не автопилот).

**Ошибки:**
- Сегментов не 3 → «Это не похоже на JWT: ожидались 3 части через точку, найдено 2».
- Сегмент не base64url → указываем **какой** («header повреждён: не base64url на позиции 14»).
- Payload — не JSON → показываем header (он валиден!) и сырой payload. ⚠️ **Частичный успех показываем частично**, а не обнуляем экран. Бывают JWT с не-JSON payload — это законно.
- `alg: "none"` → 🔴 **красный блок**: «Токен заявляет alg: none — подписи нет. Такой токен может подделать кто угодно. Если ваш сервер его принимает — это уязвимость».
- `exp` в прошлом → ⛔ + текстом «ПРОСРОЧЕН», не только цветом.
- Верификация RS/ES: ключ не парсится → «Ключ не распознан. Ожидается JWK (JSON) или PEM (-----BEGIN PUBLIC KEY-----). Приватный ключ вставлять не нужно и не следует».
- ⚠️ Алгоритм в header не совпадает с типом ключа → говорим прямо, а не «verification failed».

### 4.5 Валидация по JSON Schema

1. Таб «Схема». Документ уже разобран в табе «Данные» — берём его. Нет документа → пустое состояние со ссылкой «Сначала откройте данные».
2. Вставка/файл схемы → валидация в **Worker** (схема может быть злая: `pattern` с катастрофическим бэктрекингом → ReDoS, ср. `compose` §6.5). ⚠️ **Таймаут 5 с → `worker.terminate()`** + честное «Валидация не завершилась за 5 секунд и была прервана. Обычно причина — тяжёлый `pattern` в схеме».
3. Ошибки — списком, каждая с JSONPath, ожиданием, фактом, указателем на место **в схеме**, кнопкой «Показать в данных» (переход в таб Данные + разворот дерева до узла + подсветка).
4. Валидно → «✓ Документ соответствует схеме» + ⚠️ приписка «Проверены: типы, required, enum, pattern, диапазоны. **Не** проверены: внешние `$ref` (сети нет), `format` (выключен)».

**Ошибки:** схема не JSON → ошибка позиции, как у документа. Схема — не схема (нет `type`/`$schema` и валидные ключи не найдены) → предупреждение, но валидируем: пустая схема `{}` валидна и валидирует всё. `$ref: "https://..."` → ⚠️ **явная ошибка, а не тихий пропуск**: «Внешняя ссылка `https://…` не может быть загружена: расширение не ходит в сеть. Вставьте схему целиком (`$defs`)».

### 4.6 Конвертировать JSON → YAML

1. Документ разобран. «Конвертировать в → YAML».
2. Конвертация в Worker (для 50 МБ — обязательно), сериализация через `yaml` (ISC).
3. Сплит-вид (§2.5), панель предупреждений, `[Копировать]` / `[Скачать]` / `[Сделать YAML документом]` (последнее — заменяет документ, чтобы можно было продолжить: YAML → дерево → XML).

**Ошибки/предупреждения (обязательны, §2.5):**
- Циклов в JSON быть не может, но в XML→JSON — могут быть повторяющиеся теги → массивы. Правило схлопывания **показываем**, а не прячем.
- JSON→CSV на невплоском документе → **не «пустой CSV»**, а: «CSV — плоская таблица. Ваш документ — объект с вложенностью 4 уровня. Выберите массив объектов внутри документа: [$.users] [$.meta.items]» — с реальными кандидатами из дерева.
- JSON→XML: ключи, недопустимые как имена тегов (цифра в начале, пробел, `:`) → показываем правило подстановки и список затронутых ключей.
- YAML с якорями/алиасами (`&a`/`*a`) → в JSON они разворачиваются; при рекурсивном якоре — ⚠️ отказ с объяснением (иначе бесконечная развёртка = OOM, это billion-laughs).
- ⚠️ **XML: сущности.** `DOMParser` не раскрывает внешние сущности (XXE в браузере не работает), но «billion laughs» на внутренних — теоретически. Ограничиваем размер после развёртки, при превышении — отказ с текстом.

Это главный экран ошибки инструмента, и он должен быть лучше, чем у всех остальных.

```
┌────────────────────────────────────────────────────────────────────────────────┐
│ Формат: [JSON ▾] авто     ✗ Ошибка разбора                                     │
├────────────────────────────────────────────────────────────────────────────────┤
│   12   "roles": ["admin", "dev"],                                              │
│   13   "meta": {                                                               │
│   14     "version": "1.0",,                                                    │
│                            ^                                                   │
│                            └─ строка 14, столбец 26                            │

│                                                                                │

│   [ Убрать лишнюю запятую ]   ← правка одного символа, показываем какого       │
│   [ Разобрать как JSON5 ]     ← JSON5 допускает висячие запятые                │
│   [ Разобрать как JSONC ]                                                      │
└────────────────────────────────────────────────────────────────────────────────┘
```

Как получаем позицию: **`jsonc-parser`** (error-tolerant, отдаёт оффсеты токенов — PLAN.md (Часть II) §2.5). ⚠️ **Не** полагаемся на текст `SyntaxError` от движка: он разный в V8 и SpiderMonkey и не даёт стабильных оффсетов.

⚠️ **Частичный результат.** `jsonc-parser` умеет вернуть дерево «до ошибки» — показываем его в фоне полупрозрачным, с плашкой «показана часть до ошибки». Пустой экран при одной лишней запятой в 40 МБ — жестокость.

## 6. Честность UI — где обязаны признать ограничение

Список исчерпывающий. Каждый пункт — место, где легко соврать, и где врать нельзя.

1. **Firefox перехватывает JSON-страницы.** Отключить нельзя. Сказать прямо, дать инструкцию (§5.5).
2. **Авто-формат требует «доступа ко всем сайтам».** Не эвфемизм «расширенный режим», а точная формулировка предупреждения браузера + причина (§2.11).
3. **Точность чисел.** Если source access нет — говорим, что точность потеряна, а не показываем округлённое как исходное (§5.6).
4. **Конвертация с потерями.** JSON→CSV/XML теряет данные. Показываем **что именно** (§2.5).
5. **`exp` проверяется по часам компьютера.** Кривые часы → кривой вердикт (§2.6).
6. **«Подпись не совпала» ≠ «токен подделан».** Отличить подделку от неверного ключа невозможно (§2.7).
7. **JSON Schema: поддерживается не всё.** Внешние `$ref` — никогда (сети нет). `format` — только по опции, и это аннотация, а не ограничение по спеке (§2.8, §4.5).
8. **Автодетект формата может ошибиться.** Показываем «авто» рядом с выбранным форматом и даём override одним кликом. Не притворяемся, что уверены.
9. **Прогресс — по прочитанным байтам, а не по «проценту разбора».** Никаких выдуманных чисел (§5.1, и это буквально урок `PLAN.md` §18a).
10. **Документ > 1 МБ не сохраняется.** Сказать в Настройках, а не «потерять» при перезагрузке (§3).
11. **Подсветка отключается на очень больших документах.** Сообщить, а не тихо показать серый текст (§5.2).
12. **CSV-автодетект разделителя может ошибиться** на данных с `;` внутри кавычек. Показываем выбранный разделитель и число колонок; расхождение числа колонок между строками — предупреждение с номером строки.
13. **«Отформатировано расширением»** — на чужой странице всегда видно, что вид изменён нами, и есть `✕` (§2.12). Пользователь не должен путать наш рендер с настоящей страницей.

## 7. Безопасность

### 7.1 Рамка: JWT — это креденшл

Копия для баннера над табом JWT (финальный текст, не черновик):

> **🔒 JWT — это учётные данные. Токен даёт доступ к вашему аккаунту.**
> Расширение работает на **100% офлайн**: токен не покидает браузер, не сохраняется на диск и никуда не отправляется. Сети у расширения нет вообще — ни аналитики, ни телеметрии, ни отчётов об ошибках.
> И всё же: относитесь к токенам как к паролям. Прежде чем вставить чужой токен в **любой** онлайн-инструмент — подумайте, куда он уедет.

Для HS256 (финальный текст):

> **⚠️ HS256 проверяется общим секретом.** Секрет — это ключ, которым подписываются все ваши токены; кто его знает, тот выпускает токены от вашего имени.
> Мы держим его **только в оперативной памяти**: он не пишется в хранилище расширения, не попадает в автосохранение документа и исчезает, когда вы закрываете вкладку. Поле не автозаполняется и не проверяется орфографией.

Для верификации по публичному ключу:

> Вставьте **публичный** ключ (JWK или PEM). Приватный ключ здесь не нужен — и вставлять его не следует ни сюда, ни куда-либо ещё.

⚠️ Почему это в UI, а не только в листинге: PLAN.md (Часть II) §2.3 и §9 — «disclosures in the Chrome Web Store description do not satisfy this requirement». Мы **ничего не собираем**, поэтому формально prominent disclosure не требуется — но с 1 августа 2026 планка поднята, а рамка «100% офлайн» ещё и является продуктовым позиционированием (jwt.io делает так же).

### 7.2 Что не персистится НИКОГДА

| Что | Почему |
|---|---|
| Токен JWT (целиком, header, payload, signature) | `authenticationInfo`. Хранить = собирать |
| Секрет HS256 | То же, но хуже: это ключ выпуска |
| Публичный ключ (v1) | Не секрет, но «хранилище ключей» — другая цель. Не начинаем |
| История токенов, «недавние JWT» | 🔴 Соблазн огромный, ответ — нет |
| Содержимое таба JWT в `local:document` | ⚠️ Автосохранение документа **не должно** случайно поймать токен: таб JWT имеет **отдельный, не-персистируемый** store. Это архитектурный инвариант, а не «не забыть» |

⚠️ Дополнительно: поле секрета — `type="password"`, `autocomplete="off"`, `spellcheck="false"`, `data-1p-ignore` (менеджеры паролей не должны его подхватывать). Кнопка «👁» — только на `mousedown`/`focus`, не липкая.

### 7.3 Никакого `innerHTML` — и почему это здесь особенно важно

Мы рендерим **чужой недоверенный текст**: JSON из бага коллеги, XML со случайного сайта, токен из чужого лога. Страница расширения имеет доступ к `chrome.*` и `storage`. XSS здесь — не дефейс, а **компрометация расширения** (та же логика, что у `compose` §6.3, и она применима к нам не меньше).

- 🔴 **Zero `innerHTML` / `outerHTML` / `insertAdjacentHTML` / `document.write` в нашем коде.** Всё DOM — через React или явные DOM API. Это уже официальная позиция репозитория (`STORE.md`, ответ на AMO `UNSAFE_VAR_ASSIGNMENT`: «our own source contains zero uses of innerHTML»). Новое расширение обязано не сломать этот ответ — иначе придётся объясняться с AMO по существу, а не отпиской про React-рантайм.
- 🔴 Подсветка синтаксиса — **не через генерацию HTML-строк**, а через `Highlight API` + `Range` над плоским текстовым `<pre>` (PLAN.md (Часть II) §2.5). Это не только производительность — это **отсутствие HTML-инъекции по построению**: мы никогда не собираем разметку из пользовательского текста.
- 🔴 XML-превью — **не** `innerHTML` от строки XML. `DOMParser.parseFromString(text, 'application/xml')` → обход DOM → рендер React-узлами. ⚠️ Никогда не парсить чужой XML как `'text/html'`.
- 🔴 Никаких `eval`, `new Function`, `setTimeout(string)`. Отсюда же и `@cfworker/json-schema` вместо AJV (PLAN.md (Часть II) §2.4).
- CSP страниц расширения: `script-src 'self'; object-src 'self';` — **без** `wasm-unsafe-eval` (WASM нам не нужен) и **без** `connect-src` куда-либо.

### 7.4 Сеть

🔴 Ноль. Ни `fetch`, ни `XMLHttpRequest`, ни `WebSocket`, ни `sendBeacon`, ни `<img src="http…">`, ни внешних шрифтов/CSS. Следствия, которые надо принять как продуктовые решения:
- Нет загрузки JWKS по URL (PLAN.md (Часть II) §2.3: «Лучше без него»).
- Нет `$ref` на внешние схемы.
- Нет «поделиться ссылкой», нет «открыть в jsonhero».
- Нет Sentry. ⚠️ Ошибки видим только сами — значит, качество берётся тестами, а не телеметрией.

| **Worker завис** (злая regex в `pattern` схемы, патологический CSV) | Таймаут: парс 30 с, валидация 5 с, конвертация 30 с → `worker.terminate()` + честное сообщение с причиной и кнопкой «Повторить». ⚠️ **`terminate()` — единственный способ прервать JS-цикл.** Тот же вывод, что в `compose` §6.5 |
| **Worker умер сам** (OOM) | §5.3: фолбэк «открыть как текст» |
| **Квота `storage.local`** (~10 МБ, `unlimitedStorage` не берём) | Пишем документ только ≤1 МБ. Перед записью — `navigator.storage.estimate()`. `set()` в `try/catch`: при `QUOTA_BYTES` → баннер «Не удалось сохранить документ: хранилище браузера заполнено. Работа продолжается, но документ не восстановится после перезагрузки» + кнопка «Стереть сохранённое». ⚠️ **Никогда не падать молча** — это буквально баг `blur` из `PLAN.md` §18a |
| **Квота `storage.sync`** (8 192 Б/элемент) | Prefs — плоский объект из ~15 булевых/enum, ~300 байт. Ничего пользовательского в `sync` **никогда**. Пороговая проверка при записи: если `JSON.stringify(prefs).length > 4096` — это баг, логируем и пишем в `local` |
| **Буфер обмена не сработал** (`readText` без фокуса, `writeText` в Firefox без user activation) | Чтение: фолбэк на фокус + подсказку «⌘V» (§4.1). Запись: фолбэк — выделить текст и показать «Нажмите ⌘C». ⚠️ **Никогда не показывать «Скопировано», не проверив результат промиса.** Метка «Скопировано» сбрасывается через 1,5 с (конвенция дома, `PLAN.md` §18c) |
| **Скачивание не сработало** | `<a download>` + `URL.createObjectURL` из tool page. `revokeObjectURL` через 60 с. ⚠️ Разрешения `downloads` **не берём** (PLAN.md (Часть II) §3.1 — оно не нужно) |
| **Битый ввод любого рода** | Всегда частичный результат + позиция, никогда — пустой экран (§5.4) |
| **`permissions.request()` вернул false** | Тумблер откатывается, состояние читается из `permissions.contains()` при каждом маунте, не из настройки (§3) |
| **Разрешение отозвано извне** (chrome://extensions) | `permissions.onRemoved` → снимаем регистрацию content-script, обновляем UI. ⚠️ Иначе UI показывает «включено», а фича мертва |
| **Настройки не прочитались** | Контролы disabled до первого чтения; при ошибке — «Повторить». ⚠️ Не рендерить дефолты как «текущее» — первый клик их зафиксирует, затерев настоящие |
| **Гонка записи настроек** (RMW) | Очередь записи / `navigator.locks` — уже отработанный в `blur` (§18c) приём |

## 9. Доступность и тема

### 9.1 Тема (конвенция дома — не изобретать)

- Токены — из `packages/ui` (⚠️ **пакета ещё нет**; PLAN.md (Часть II) §7: «`packages/ui` перестал быть опциональным… делаем **до** старта новой волны»). До его появления — копия набора из `extensions/seo/entrypoints/popup/style.css`, но это долг, а не решение.
- `:root { color-scheme: light dark; }` + `@media (prefers-color-scheme: dark)` + `:root[data-theme='light'|'dark']` с явным `color-scheme` внутри — точно как в `seo`. Атрибутный селектор (0,2,0) перебивает медиа-фолбэк (0,1,0).
- Антифликер: синхронный сид темы из `localStorage` **до** `createRoot` (`seedTheme()` в `seo/utils/theme.tsx`). Для tool page это критичнее, чем для попапа: белая вспышка на весь экран.
- ⚠️ `color-scheme` обязателен, иначе `<select>`, скроллбары и `input[type=password]` останутся светлыми в тёмной теме (баг, уже исправленный в `PLAN.md` §18a).
- ⚠️ **`::highlight()` и тема.** Цвета подсветки — тоже токены. И: в `::highlight()` **нельзя менять `font-weight`** (PLAN.md (Часть II) §2.5) — значит, ключи от значений отличаем **цветом + они и так синтаксически различимы кавычками и двоеточием**. Никакого «жирные ключи» в дизайне. ⚠️ `forced-colors: active` (Windows High Contrast) отключает пользовательские цвета — подсветка исчезнет; текст обязан остаться читаемым и структура понятной без неё.

### 9.2 Не только цветом

| Сигнал | Цвет | + текст/знак |
|---|---|---|
| Документ валиден | зелёный | `✓ Валиден` |
| Ошибка разбора | красный | `✗ Ошибка разбора` + позиция |
| Подпись верна / неверна | зелёный/красный | `✓ ПОДПИСЬ ВЕРНА` / `✗ ПОДПИСЬ НЕ СОВПАДАЕТ` |
| Токен просрочен | красный | `⛔ ПРОСРОЧЕН на 1 ч 4 мин` |
| Предупреждение конвертации | жёлтый | `⚠ 3 предупреждения` |
| Типы в дереве | цвет | `{}` `[]` `"` `#` `T/F` `∅` — типографские маркеры, а не только цвет |

### 9.3 Клавиатура

Инструмент разработчика без клавиатуры — не инструмент.

| Клавиша | Действие |
|---|---|
| `Tab` / `Shift+Tab` | Переходы между регионами. ⚠️ Дерево — **один** таб-стоп (`roving tabindex`), а не 40 000 |
| `↑` `↓` | Строка дерева вверх/вниз |
| `→` `←` | Раскрыть / свернуть (и уйти к родителю на закрытом узле) — семантика `tree` из WAI-ARIA APG |
| `Home` / `End` | Начало/конец дерева |
| `Enter` | Показать значение в инспекторе |
| `⌘/Ctrl+C` | Копировать значение фокусного узла |
| `⌘/Ctrl+⇧+C` | Копировать JSONPath фокусного узла |
| `⌘/Ctrl+F` | Поиск (⚠️ на >50 МБ — отключён, с объяснением) |
| `⌘/Ctrl+⇧+F` | Beautify |
| `Esc` | Закрыть `<dialog>` / снять фокус с поиска |
| `1` `2` `3` | Переключение табов (когда фокус не в поле ввода) |

- Роли: дерево — `role="tree"` / `treeitem` / `group`, `aria-expanded`, `aria-level`, `aria-setsize`, `aria-posinset`. Табы — `role="tablist"`/`tab`/`tabpanel` с `aria-controls` **только на активном** (баг, уже исправленный в `blur`, §18c).
- `aria-live="polite"` — статус парсинга, число ошибок валидации, «Скопировано», результат верификации. ⚠️ `aria-live="assertive"` — **только** на ошибку разбора и на «подпись не совпала».
- `:focus-visible` — `outline: 2px solid var(--accent); outline-offset: 2px` на **всём** интерактивном (WCAG 2.4.7). Уже есть в стилях дома.
- Тач-таргеты ≥ 44px на узком лейауте (Firefox Android — целевая платформа, PLAN.md (Часть II) §8).
- `prefers-reduced-motion` — спиннер замедляется, View Transitions отключаются целиком.
- ⚠️ Виртуализация ломает встроенный поиск браузера (`Ctrl+F` найдёт только окно). Свой поиск — обязателен, и **сказать**, что браузерный не увидит всё.

## 12. Найденные противоречия и недоспецифицированное (в Части II / Части I)

1. ⚠️ **Разрешения vs фича.** PLAN.md (Часть II) §2.2 объявляет baseline `storage` + `contextMenus` + `activeTab` **и одновременно** обещает контекстное меню «Format JSON on this page». Инъекция в страницу требует `chrome.scripting` (в MV3 `activeTab` даёт **хост**, но не право звать `scripting.*`). Разрешено это противоречие в §4.3: `scripting` уходит в **`optional_permissions`** и запрашивается permission-only (без host → **без страшного предупреждения**), а baseline-манифест остаётся ровно тем, что заявлен. ⚠️ На Firefox MV2 `scripting` вообще не нужен (`tabs.executeScript` от `activeTab`) — там кнопка активна сразу. **Это надо провалидировать на живой сборке до старта.**
2. ⚠️ **`packages/ui` не существует.** PLAN.md (Часть II) §7 объявляет его блокером волны («делаем **до** старта»), но в репо только `packages/core`. `devdata` не может быть первым, кто «просто скопирует токены» — иначе долг из `PLAN.md` §18 растёт третий раз.
3. ⚠️ **Side panel не решён.** PLAN.md (Часть II) §11 держит `side_panel` vs `sidebar_action` открытым вопросом для `compose`, но не упоминает `devdata`. Здесь он решён отсечением: v1 без side panel.
4. ⚠️ **Квота `storage.local` vs «50 МБ JSON».** PLAN.md (Часть II) обещает 50 МБ документы и «локальную историю буферов» (§10.1: «Локальная история — можно»), но `storage.local` — ~10 МБ, а `unlimitedStorage` в baseline нет. Противоречие снято ограничением «персистим только ≤1 МБ» + честной надписью. Если история буферов вернётся в v2 — это IndexedDB, а не `storage`.
5. ⚠️ **PLAN.md (Часть II) §10.1 не даёт таба «Схема» отдельного статуса**, хотя валидация по схеме — v1. Здесь она сделана табом (документ + схема = два ввода, в один экран «Данных» не влезает).
6. ⚠️ **Не сказано, что делать с CSV как с *документом*.** CSV — таблица, а не дерево. Дизайн-решение здесь: CSV показывается таблицей (виртуализованной), дерево для него не строится, и это состояние надо явно поддержать в UI.
